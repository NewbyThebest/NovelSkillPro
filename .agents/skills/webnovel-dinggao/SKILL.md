# 定稿回写

## 目标与路径

用户确认章节定稿后，先确认最新正文与章纲的关系，再将这份正文作为后续维护的唯一事实来源。完成本章归档、控制台更新和必要的规划维护后，等待用户进入下一章，不自动继续写作。

- 将本技能目录记为 `SKILL_DIR`，包含 `.agents/` 的项目根目录记为 `WORKSPACE_ROOT`。
- 小说文件从 `WORKSPACE_ROOT` 读取和写入；模板从 `<SKILL_DIR>/references/` 的实际绝对路径读取。
- 已确认的定稿、控制台状态和已回写章纲不得被后续草稿覆盖。

## 步骤 0：定位章节与定稿前对齐

1. 结合用户明确说明、当前对话、`4-正文/` 中的 `第X章_草稿_[标题].md` 或 `第X章_定稿_[标题].md`、以及 `控制台/进度.md` 定位待处理章节。
   - 上下文、文件和进度能指向同一章时继续。
   - 存在多个候选、章节号冲突或用户未说明时，停止并只问“要为第几章定稿？”。
2. 长篇模式按本章章号找到 `3-大纲/` 下对应的十章章纲文件。第 1—10 章使用 `章纲_1-10章.md`，第 11—20 章使用 `章纲_11-20章.md`，后续依此类推；文件名固定，不根据剧情目标或实际收束情况调整。`SHORT_STORY_MODE` 下读取 `3-大纲/短篇章纲.md`。读取其中本章章纲与待定稿正文。
3. 在重命名前做只读对齐：确认正文是否改变了本章核心事件、角色状态、伏笔、结尾状态或设定事实。
   - 无实质差异，或用户明确以当前正文为准时，继续转正。
   - 存在实质差异时，向用户列出“章纲 vs 正文”的事实差异，并让用户选择：按正文回写章纲，或先返回草稿修正。选择修稿时不转正、不更新控制台。
   - 本章存在“待定稿同步项”时，同时核对定稿正文是否实际保留了这些差异。已经没有进入正文的差异直接删除；仍然成立的差异留到步骤 4 处理，不在转正前擅自修改设定或总纲。
4. 本章涉及主要角色或核心关系时，读取 `2-设定/2.3_主要角色设定表.md` 或 `2-设定/2.3.2_核心关系.md` 的相关条目；出现、复用或结算阶段角色时，读取 `2-设定/2.3.1_阶段与功能角色记录.md` 的相关条目，判断正文产生的是短期状态还是稳定档案变化。当前文件的低频归档索引指向本章复用角色时，再读取对应 `2-设定/角色归档/[阶段名].md` 条目，不加载无关归档。
5. 用户确认以该正文定稿后，将 `第X章_草稿_[标题].md` 重命名为 `第X章_定稿_[标题].md`，只替换状态字段并保留章节标题。标题在定稿前已经变更时，使用用户最新确认的安全文件名标题，不保留旧标题副本；用户已手动转正时直接使用定稿文件。正文内容从故事第一句开始，不在文件内重复章号或标题。
6. 若该章已经是定稿，且 `控制台/进度.md` 已记录为完成，默认视为已收尾，不重复更新。只有用户明确要求修复、补回写或重新结算时才继续。

## 步骤 1：回写本章章纲

以定稿正文为准更新本章章纲的事实描述：剧情骨架、读者回报或蓄势、伏笔线索和结尾状态。

- 回写剧情骨架时，定稿正文只用于确认已发生的事实，不按正文段落、动作链或场景镜头复述。
- 剧情骨架必须使用 Markdown 有序列表。每个列表项对应一次独立的局面变化，只写发生什么、谁作何选择、造成什么结果。
- 同一场连续行动中，若中途没有新的选择、信息或局面变化，应合并为同一项，不按动作先后拆成多项。
- 不回填具体台词、连续动作、招式过程、走位、现场反应、环境描写或情绪节拍，除非该细节本身改变后续剧情边界。
- 定稿正文与剧情骨架只需在事件顺序、人物选择、结果和已确认事实上一致。
- 只更新已被用户确认接受的正文差异，不反向改写定稿正文。
- 本章章纲缺失时，基于定稿正文补建最小可用记录，不补造未写出的剧情。
- 本章存在“创作原料（定稿前临时保留）”时，只有在正文转正且事实章纲回写完成后才删除该节。草稿生成、审查、润色或用户尚未确认定稿时不得提前清理。
- 没有写进定稿正文的创作原料默认不成为事实。用户明确要求留到后续时，先确认去向，再转入下一章创作原料或尚未写出的规划；不得自动写入控制台、角色档案或伏笔记录。
- “待定稿同步项”不随创作原料一起自动删除。只有对应差异未进入定稿正文，或已经按步骤 4 完成全局同步后，才清理对应条目；用户暂缓全局修订时保留该节。
- 回写完成后说明本章章纲是否有改动，以及改动的事实范围。

## 步骤 2：更新控制台

首次定稿时，若控制台文件缺失，从 `<SKILL_DIR>/references/` 复制对应模板到 `控制台/`，并删除模板示例后再填写真实项目事实。

每次定稿后按以下规则更新：

| 文件 | 更新规则 |
| :--- | :--- |
| `控制台/进度.md` | 每章必更：当前承接、最近三章、最新定稿复盘、下一步与阶段缺口、压缩后的里程碑。 |
| `控制台/角色状态.md` | 按需更新当前活跃角色的动态快照；没有变化且仍属当前工作集时保留，不再影响当前阶段创作时删除。 |
| `控制台/伏笔追踪.md` | 新埋、推进、收回或失效的伏笔和关系线发生变化时更新；没有变化时保留原记录。 |

所有记录只写正文已经发生的事实，不把下一章计划写成已发生状态，不把猜测写成伏笔结论。完成后明确列出本次实际更新的控制台文件和条目。

### 角色状态文件维护规则

`控制台/角色状态.md` 是下一章直接读取的动态工作表，不是已登场角色名册或逐章档案。始终按 `<SKILL_DIR>/references/角色状态模板.md` 的结构维护：

1. 只保留当前场景、下一章承接或当前阶段目标与约束仍需直接读取的角色。核心角色暂时离场，但其状态仍构成当前阶段目标或限制时，保留压缩快照。
2. 位置、伤势、目标、资源、能力、认知和关系均覆盖更新，不在旧内容后追加章节经过。
3. 每个角色使用且只使用一个“最近更新依据”，只填写最后一次实际改变其状态的定稿章节；不得保留“本章变化依据”或列出多个历史章节。
4. 稳定身份、外貌、能力规则和写作边界写入角色设定；长期未决问题写入伏笔追踪；逐章过程写入定稿正文和十章章纲，不在状态文件复制。
5. 角色死亡、永久离场、完成场景作用，或其状态已不影响当前阶段创作时，删除对应条目。可能复用的阶段角色由低频角色归档保存最后确认状态，重新进入当前规划时再补回状态条目。
6. 每章只处理本章发生变化或需要退出工作集的角色；第 10、20、30 章等十章节点和阶段切换时复查全部状态条目。

更新完成后运行：

```bash
python <SKILL_DIR>/scripts/validate_character_state.py <WORKSPACE_ROOT>
```

校验未通过时先修正角色状态文件再继续。脚本不可用时，至少人工确认：没有“本章变化依据”、每个角色恰有一个“最近更新依据”、该字段只含一个章节号、其他状态字段没有逐章历史、文件中没有已经退出当前工作集的角色。

### 进度文件维护规则

`控制台/进度.md` 是供下一章直接使用的动态工作台，不是逐章档案。始终按 `<SKILL_DIR>/references/进度模板.md` 的五段结构维护：

1. **当前承接**：记录当前章节、故事时间、地点、章尾局面和即时未决。无法确认具体时间、位置或身份时，写“具体未知”或“最后确认于”，不能把角色判断写成客观事实。
2. **近期情节流**：只保留最新三章定稿摘要，不足三章时保留已有全部。每章一至两句，只写影响下一章承接的选择、结果和状态变化；加入新章时删除最早一章，更早历史由对应十章章纲保存。
3. **最新定稿复盘**：只记录最新章的字数与状态、章纲差异、待同步项和定稿边界。没有差异或待同步项时直接写“无”，不累积历章复盘。
4. **下一步与阶段缺口**：长篇的当前阶段名称必须与总纲完全一致，`SHORT_STORY_MODE` 照录短篇方向中的当前推进段；下一章核心任务只写一个。近期行动保留一至三项，阶段完成缺口不超过三项，持续约束不用复选框。
5. **里程碑日志**：只记录不可逆状态变化、阶段结果、关键决策、核心能力链成形或全项目审计等跨章成果。不得写“第X章定稿”或逐章剧情复述；同一事件链合并成一条，通常保留五至十条。阶段结束时，将该阶段多条里程碑压缩为一条阶段结果。

每章定稿后立即删除已经完成的待办。确有跨章价值的完成项转为里程碑，其余直接删除。长期谜团与未收伏笔只在 `控制台/伏笔追踪.md` 维护，角色的伤势、位置、认知与关系只在 `控制台/角色状态.md` 维护，逐章历史只在十章章纲维护；进度文件不复制这些长记录。

更新完成后运行：

```bash
python <SKILL_DIR>/scripts/validate_progress.py <WORKSPACE_ROOT>
```

校验未通过时先修正进度文件再交付。脚本不可用时，至少人工确认：近期情节恰为最新三章、下一步没有 `[x]`、里程碑没有逐章定稿记录、当前章节与字数匹配最新定稿、当前阶段名称与对应的长篇总纲或短篇方向一致。

角色档案按以下规则处理：

- 当前位置、短期伤势、临时资源、即时关系状态和当章认知只写入控制台，不回写角色设定文件。
- 新出现的一次性人物在本章后仍会影响当前阶段或需要复用，并形成稳定外貌、能力、场景功能、最新结果或写作边界时，才补入 `2-设定/2.3.1_阶段与功能角色记录.md`。人物在同章完成全部作用且没有明确复用价值时不建档。
- 已有阶段角色出现不可逆结局或稳定设定变化时，覆盖更新其最新结果。“当前得失”和“当前场景结果”不得按章节追加经过，逐章历史只留在定稿正文和十章章纲。
- 不对应固定个人的岗位、群体和组织规则移入 `2-设定/2.1_新书设定案.md`，不继续作为角色条目维护。
- 阶段角色开始拥有独立长期目标、反复影响主线或形成持续关系时，将条目移入 `2-设定/2.3_主要角色设定表.md`，并从阶段角色文件删除旧条目，不保留两份版本。
- 阶段角色已经死亡、永久离场或完成全部作用，且没有明确复用价值时，从阶段角色文件删除。删除前确认其结果已经进入定稿正文和十章章纲，不另建完整历史档案。
- 角色确有复用可能、但当前阶段近期不再使用时，将稳定事实、最后确认状态、复用边界和事实来源移入 `2-设定/角色归档/[阶段名].md`，阶段角色文件只保留一行归档索引。角色重新进入当前规划时，将完整条目移回并删除归档旧条目。
- 每章只处理本章涉及的阶段角色；第 10、20、30 章等十章节点和阶段切换时复查阶段角色文件全部条目，逐一判断保留、升格、归入设定、删除或低频归档。
- 只有关系双方均已有 `2-设定/2.3_主要角色设定表.md` 中的角色条目，且已确认的核心关系基础或边界发生实质变化时，才更新 `2-设定/2.3.2_核心关系.md`。阶段与功能角色、归档角色和未建档人物不得在此建立条目；主要角色表中尚未形成核心关系的人物不补空白条目。尚未确认的未来关系只写入未写规划，不当成角色事实。

## 步骤 3：十章章纲范围（仅长篇）

长篇章纲按章节号固定为十章一个文件，不判断范围目标，不使用 `x`，也不因剧情实际收束而改名。

- 本章定稿后只回写其所属的十章章纲文件。
- 本章为第 10、20、30 章等范围末章时，文件保持原名和原有范围，不创建下一份章纲；但必须进入步骤 4 的十章节点全项目一致性审计。完成审计不等于必须修订规划。
- 下一章进入新的十章范围时，由 `webnovel-zhengwen` 创建对应的 `章纲_[起始章]-[结束章]章.md`，并写入新的“当前起点”。

短篇模式跳过本步骤。

## 步骤 4：一致性审计与按需规划修订

不在每章自动改总纲或设定。以下情况才触发复盘：

- 本章有仍然成立的“待定稿同步项”。
- 当前阶段的完成标志已经达成。
- 定稿正文持续偏离当前总纲的阶段方向、读者承诺或硬约束。
- 本章为第 10、20、30 章等固定十章节点。此时强制执行全项目一致性审计，但只有审计发现需要调整时才进入规划修订。
- 用户明确要求核对正文与预期、设定、大纲、控制台之间的出入。
- 用户明确要求调整后续规划。

### 十章节点全项目一致性审计

第 10、20、30 章等固定十章节点定稿后，必须同时执行事实一致性审计与剧情方向审计，不能只检查十章章纲。用户在非十章节点明确要求全面检查时也执行本节；用户未指定范围时，核对当前全部定稿正文。

#### 审计范围与事实源

1. 读取刚完成十章范围内的全部定稿正文，以正文中实际发生的内容作为事实源。当前文件引用了更早事实时，继续读取对应的早期定稿章节，不能凭控制台或记忆反推正文。
2. 读取当前项目中的全部 `1-预期/*.md`、`2-设定/*.md`、`3-大纲/*.md` 和 `控制台/*.md`。不读取旧流程备份、临时目录或技能镜像作为小说事实。
3. 草稿、创作原料和未来章纲只代表候选或计划，不能覆盖定稿正文。未写规划只检查起点、前提和因果是否仍然成立，不能当成已经发生。
4. `2.4_文风.md` 只核对其中声称已经稳定形成的长期写法，不在本步骤进行逐句文风审稿。

#### 事实一致性检查

逐项检查以下内容，不得只搜索同名关键词：

1. 人物身份、称呼、位置、伤势、能力、资源、认知、目标和关系是否与定稿正文一致。
2. 世界规则、组织权限、力量体系、核心机制的触发条件、能力边界和代价是否被正文新增事实推翻或补充。
3. 已发生事件的时间、地点、因果、选择、结果和存亡状态是否在章纲与控制台中准确回写。
4. 正文已经新增或推进的角色、能力、关系、伏笔和阶段事实是否存在漏记；旧状态、旧字数、旧章节号、过期待办和临时区块是否仍未清理；角色状态文件是否混入已退出当前工作集的人物或累计逐章依据；阶段角色文件是否混入已结算人物或在最新结果中累计逐章历史，活跃条目与低频归档是否重复，归档索引是否指向实际条目。
5. 是否把猜测、梦境、谎言、角色误判、未确认身份或未来计划误记为客观事实；也要检查已经确认的事实是否仍被写成未知。
6. 同一事实在预期、设定、大纲和控制台中是否出现互相冲突的并行版本，未写章纲的入口状态是否仍能承接最新定稿正文。

#### 剧情方向检查

1. 当前阶段的完成标志是否全部达成，不能只看章节数量或单个高潮是否结束。
2. 最近十章的实际路线是否仍在推进当前阶段目标，还是已经形成持续偏移、替代路线或新的主要矛盾。
3. 核心读者承诺、人物目标、关键关系与硬约束是否仍与总纲一致。
4. 长期未推进的关键任务或伏笔是否已经影响阶段成立，需要调整近端规划或阶段表述。

#### 处理审计结果

- 控制台或已定稿章节章纲存在明确漏记、旧状态或与正文一一对应的记录错误时，直接按定稿正文修正，并在交付中列明。
- 预期、稳定设定、总纲、未写规划与正文冲突，或修正需要追溯改动定稿正文时，先列出“定稿事实 / 冲突文件与原记录 / 影响 / 建议处理”，等待用户确认；不能静默覆盖。
- 不能确认哪一份内容正确时，保留不确定性并报告问题，不自行补设定。
- 审计没有发现问题时，在收尾交付中明确写出“全项目一致性审计完成，未发现需要修订的出入”及核对范围。
- 审计发现阶段完成、持续偏移或近端安排已不适用时，另行输出“原计划 vs 实际走向 vs 后续影响”，等待用户确认后再修订规划。

短篇模式默认跳过十章节点审计，除非用户明确要求全面检查或调整尚未写的结局与方向。长篇在用户确认审计中需要修订的方案后，按下列分工执行：

- **待定稿同步**：逐条说明定稿事实与现有设定、总纲或未写规划的差异。用户确认同步后，按影响范围执行 `webnovel-setting-sync`、更新总纲或修订未来章纲；用户暂缓时保留待定稿同步项，不静默改动全局文档。

- **近端调整**：只改未写的十章章纲或未写章节的章纲，不改已定稿事实。
- **当前阶段调整**：阶段尚未完成时，只更新 `3-大纲/3.1_全书结构总纲.md` 的当前阶段或入口承诺，不补写未来阶段。
- **阶段切换**：当前阶段已经达到完成标志时，先按维护规则清理角色状态和阶段角色工作表，只保留明确带入下一阶段的动态状态与角色，再向用户说明实际收束结果并执行 `webnovel-outline`。由该技能归档已完成阶段摘要，并根据最新定稿事实共创下一个当前阶段；本技能不能自行推演并写入下一阶段。
- **设定事实调整**：先执行 `webnovel-setting-sync`，处理角色、规则、核心机制或世界设定的跨文件一致性，再回写受影响的大纲。

未触发时跳过本步骤，不输出虚构的“复盘结果”。

## 步骤 5：按需文风进化

只有用户主动要求，或已积累多章用户认可的定稿正文并出现稳定模式时，才处理文风进化。单章临时写法、一次性情绪场景和偶然用词不能作为文风规则。

1. 选取多章定稿正文，提炼稳定的叙事视角与口吻、句段节奏、场景与信息释放、对白差异、情绪表达或本书特有避免项。
2. 先向用户展示“观察到的稳定特征”和对 `2-设定/2.4_文风.md` 的拟改动；用户确认后再更新。
3. 更新时优先合并或替换原有表述，不能只在原文后持续追加。每个维度保持两至三条高优先级倾向，不放示例句，不重复 `AGENTS.md` 的硬规则。
4. 只有跨多章反复稳定的特征才同步到对应维度；项目已有“进化记录”时，只追加简短的来源范围和变更摘要，不把记录扩写成第二份文风规则。

## 收尾交付

告知用户：本章已定稿、章纲是否已对齐、控制台实际更新项、进度与角色状态校验结果、本章所属十章章纲文件，以及是否触发全项目一致性审计、规划或文风维护。触发审计时必须交代核对范围、发现的问题、已直接修正项和等待确认项。之后等待用户要求下一章、调整设定或复查，不自动继续写作。
