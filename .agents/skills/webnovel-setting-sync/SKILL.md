# 设定同步

## 职责边界

本技能维护“接下来应按什么设定写”，处理作者确认的设定变更及其跨文件影响。

- 章节正文已经定稿后产生的实际角色状态、伏笔、进度和章纲事实，由 `webnovel-dinggao` 维护。
- 本技能不重命名草稿、不将草稿转正、不替代每章定稿结算，也不自动改写已定稿正文。
- 默认把新想法视为**未来生效**。只有用户明确接受追溯修订时，才修改控制台中的当前事实或标记已定稿章节的影响。

## 路径与输入

- 将本技能目录记为 `SKILL_DIR`，包含 `.agents/` 的项目根目录记为 `WORKSPACE_ROOT`。
- 先读取用户提出的变更和相关原文件；按需读取 `2-设定/`、`3-大纲/3.1_全书结构总纲.md`、未写的批次章纲，以及控制台文件。
- 缺少现有设定文件时，说明无法“同步”什么；不要借同步之名凭空建立整套新设定。

## 步骤 1：确认变更与生效范围

先从用户已有描述中提取“改什么、改成什么、从何时生效”。只有这些信息仍不清楚时，问一次关键问题；不要固定追问原因和变更等级。

将变更归为以下一种或多种：

1. **未来扩展**：新增后续角色、规则、地图、关系或机制，不否定已写事实。
2. **当前设定修正**：修正尚未写入正文但已在设定或大纲中存在的错误。
3. **追溯修订**：新设定与已定稿正文、已回写章纲或当前控制台事实冲突。

默认按未来扩展处理。追溯修订必须由用户明确确认，不能因为“这样更爽”或“更好写”就自行追溯。

## 步骤 2：分析影响范围

读取相关文件后，列出实际受影响项，而不是套用固定清单：

| 变更内容 | 可能更新的位置 | 处理原则 |
| :--- | :--- | :--- |
| 核心机制、能力、资源或规则 | `2-设定/2.2_金手指设定.md`、总纲、未写章纲 | 写清规则和边界；已定稿章节只标记影响。 |
| 角色目标、身份、关系或新核心角色 | `2-设定/2.3_主要角色设定表.md`、总纲、未写章纲 | 只补与主线相关的信息；不为功能角色过度建档。 |
| 世界、行业、地图或社会规则 | `2-设定/2.1_新书设定案.md`、总纲、未写章纲 | 说明新规则怎样影响已有选择。 |
| 主线、阶段方向或读者承诺 | 总纲、未写批次锚点和章纲 | 不改已定稿事实，先重排未来因果。 |
| 未来伏笔或关系计划 | 未写章纲、总纲 | 未在正文出现前，不写入“已埋伏笔”或当前关系状态。 |
| 追溯修订 | 相关设定、总纲、控制台、受影响的定稿章节清单 | 先取得用户确认，再同步当前真相并标记需人工修订的正文。 |

输出简洁的影响摘要：已确认事实、拟变更事实、受影响文件、已定稿内容风险和建议生效范围。

## 步骤 3：冲突检查与确认

检查新设定是否与世界规则、角色利益、核心机制限制、总纲方向、未写章纲或已定稿事实冲突。

- 没有冲突时，说明将更新哪些文件。
- 存在冲突时，列出具体冲突与可选处理：保留旧设定、以新设定覆盖、调整新设定使其兼容，或将其限定为未来条件变化。
- 涉及已定稿正文时，明确哪些章节需要作者决定是否重写；本技能只标记影响，不直接改正文。

得到用户对变更内容和影响范围的确认后，才写入文件。

## 步骤 4：按依赖顺序同步

1. 先更新 `2-设定/` 中的真相源，保留与本书实际有关的规则、角色动机、关系边界或机制限制。
2. 再更新总纲和未写批次锚点/章纲，使未来剧情按新事实推进。
3. 默认不动控制台。只有追溯修订被用户确认，且它改变当前已经成立的角色状态、伏笔状态或关系状态时，才更新相应控制台条目，并注明这是已确认的追溯调整。
4. 已定稿正文只列入“受影响章节”，不自动改写。用户决定重写时，回到正文或润色流程处理。

不要为了留下痕迹而给每份设定文件强行添加修改记录。用户已有记录区或明确要求版本记录时，再以简短事实补充。

## 步骤 5：验证与交付

完成后重新检查：

- 设定真相源、总纲和未写规划是否使用同一套新事实。
- 追溯修订是否只改到了用户确认的范围。
- 已定稿正文是否仅被标记、没有被静默改写。
- 控制台是否只记录已经发生或已确认追溯的事实。

交付时列出实际更新文件、未更新但受影响的文件、以及下一步应回到正文、定稿回写还是等待后续规划。不要自动继续写作或结算章节。
