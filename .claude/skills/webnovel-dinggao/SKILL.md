# 定稿回写

## 目标与路径

用户确认章节定稿后，先确认最新正文与章纲的关系，再将这份正文作为后续维护的唯一事实来源。完成本章归档、控制台更新和必要的规划维护后，等待用户进入下一章，不自动继续写作。

- 将本技能目录记为 `SKILL_DIR`，包含 `.agents/` 的项目根目录记为 `WORKSPACE_ROOT`。
- 小说文件从 `WORKSPACE_ROOT` 读取和写入；模板从 `<SKILL_DIR>/references/` 的实际绝对路径读取。
- 已确认的定稿、控制台状态和已回写章纲不得被后续草稿覆盖。

## 步骤 0：定位章节与定稿前对齐

1. 结合用户明确说明、当前对话、`4-正文/` 中的 `第X章_草稿_[标题].md` 或 `第X章_定稿_[标题].md`、以及 `控制台/进度.md` 定位待处理章节。
   - 上下文、文件和进度能指向同一章时继续。
   - 存在多个候选、章节号冲突或用户未说明时，停止并只问“要为第几章定稿？”。
2. 长篇模式按本章章号找到 `3-大纲/` 下对应的十章章纲文件。第 1—10 章使用 `章纲_1-10章.md`，第 11—20 章使用 `章纲_11-20章.md`，后续依此类推；文件名固定，不根据剧情目标或实际收束情况调整。`SHORT_STORY_MODE` 下读取 `3-大纲/短篇章纲.md`。读取其中本章章纲与待定稿正文。
3. 在重命名前做只读对齐：确认正文是否改变了本章核心事件、角色状态、伏笔、结尾状态或设定事实。
   - 无实质差异，或用户明确以当前正文为准时，继续转正。
   - 存在实质差异时，向用户列出“章纲 vs 正文”的事实差异，并让用户选择：按正文回写章纲，或先返回草稿修正。选择修稿时不转正、不更新控制台。
   - 本章存在“待定稿同步项”时，同时核对定稿正文是否实际保留了这些差异。已经没有进入正文的差异直接删除；仍然成立的差异留到步骤 4 处理，不在转正前擅自修改设定或总纲。
4. 本章涉及主要角色或核心关系时，读取 `2-设定/2.3_主要角色设定表.md` 或 `2-设定/2.3.2_核心关系.md` 的相关条目；出现、复用或结算阶段角色时，读取 `2-设定/2.3.1_阶段与功能角色记录.md` 的相关条目，判断正文产生的是短期状态还是稳定档案变化。
5. 用户确认以该正文定稿后，将 `第X章_草稿_[标题].md` 重命名为 `第X章_定稿_[标题].md`，只替换状态字段并保留章节标题。标题在定稿前已经变更时，使用用户最新确认的安全文件名标题，不保留旧标题副本；用户已手动转正时直接使用定稿文件。正文内容从故事第一句开始，不在文件内重复章号或标题。
6. 若该章已经是定稿，且 `控制台/进度.md` 已记录为完成，默认视为已收尾，不重复更新。只有用户明确要求修复、补回写或重新结算时才继续。

## 步骤 1：回写本章章纲

以定稿正文为准更新本章章纲的事实描述：剧情骨架、读者回报或蓄势、伏笔线索和结尾状态。

- 回写剧情骨架时，定稿正文只用于确认已发生的事实，不按正文段落、动作链或场景镜头复述。
- 剧情骨架必须使用 Markdown 有序列表。每个列表项对应一次独立的局面变化，只写发生什么、谁作何选择、造成什么结果。
- 同一场连续行动中，若中途没有新的选择、信息或局面变化，应合并为同一项，不按动作先后拆成多项。
- 不回填具体台词、连续动作、招式过程、走位、现场反应、环境描写或情绪节拍，除非该细节本身改变后续剧情边界。
- 定稿正文与剧情骨架只需在事件顺序、人物选择、结果和已确认事实上一致。
- 只更新已被用户确认接受的正文差异，不反向改写定稿正文。
- 本章章纲缺失时，基于定稿正文补建最小可用记录，不补造未写出的剧情。
- 本章存在“创作原料（定稿前临时保留）”时，只有在正文转正且事实章纲回写完成后才删除该节。草稿生成、审查、润色或用户尚未确认定稿时不得提前清理。
- 没有写进定稿正文的创作原料默认不成为事实。用户明确要求留到后续时，先确认去向，再转入下一章创作原料或尚未写出的规划；不得自动写入控制台、角色档案或伏笔记录。
- “待定稿同步项”不随创作原料一起自动删除。只有对应差异未进入定稿正文，或已经按步骤 4 完成全局同步后，才清理对应条目；用户暂缓全局修订时保留该节。
- 回写完成后说明本章章纲是否有改动，以及改动的事实范围。

## 步骤 2：更新控制台

首次定稿时，若控制台文件缺失，从 `<SKILL_DIR>/references/` 复制对应模板到 `控制台/`，并删除模板示例后再填写真实项目事实。

每次定稿后按以下规则更新：

| 文件 | 更新规则 |
| :--- | :--- |
| `控制台/进度.md` | 每章必更：当前章节、故事时间地点、近期情节流、完成状态和下一章待办。 |
| `控制台/角色状态.md` | 角色的位置、伤势/状态、目标、资源、能力或关系发生变化时更新；没有变化时保留原记录。 |
| `控制台/伏笔追踪.md` | 新埋、推进、收回或失效的伏笔和关系线发生变化时更新；没有变化时保留原记录。 |

所有记录只写正文已经发生的事实，不把下一章计划写成已发生状态，不把猜测写成伏笔结论。完成后明确列出本次实际更新的控制台文件和条目。

角色档案按以下规则处理：

- 当前位置、短期伤势、临时资源、即时关系状态和当章认知只写入控制台，不回写角色设定文件。
- 新出现的一次性人物形成稳定外貌、能力、场景功能、当前结果或写作边界时，补入 `2-设定/2.3.1_阶段与功能角色记录.md`。
- 已有阶段角色出现不可逆结局或稳定设定变化时，更新其阶段角色条目。
- 阶段角色开始拥有独立长期目标、反复影响主线或形成持续关系时，将条目移入 `2-设定/2.3_主要角色设定表.md`，并从阶段角色文件删除旧条目，不保留两份版本。
- 已确认的核心关系基础或边界发生实质变化时，更新 `2-设定/2.3.2_核心关系.md`；尚未确认的未来关系只写入未写规划，不当成角色事实。

## 步骤 3：十章章纲范围（仅长篇）

长篇章纲按章节号固定为十章一个文件，不判断范围目标，不使用 `x`，也不因剧情实际收束而改名。

- 本章定稿后只回写其所属的十章章纲文件。
- 本章为第 10、20、30 章等范围末章时，文件保持原名和原有范围，不创建下一份章纲；但必须进入步骤 4 的十章节点周期审计。周期审计不等于必须修订规划。
- 下一章进入新的十章范围时，由 `webnovel-zhengwen` 创建对应的 `章纲_[起始章]-[结束章]章.md`，并写入新的“当前起点”。

短篇模式跳过本步骤。

## 步骤 4：按需复盘与规划修订

不在每章自动改总纲或设定。以下情况才触发复盘：

- 本章有仍然成立的“待定稿同步项”。
- 当前阶段的完成标志已经达成。
- 定稿正文持续偏离当前总纲的阶段方向、读者承诺或硬约束。
- 本章为第 10、20、30 章等固定十章节点。此时强制执行周期审计，但只有审计发现需要调整时才进入规划修订。
- 用户明确要求调整后续规划。

### 十章节点周期审计

第 10、20、30 章等固定十章节点定稿后，必须读取当前阶段、该十章章纲、控制台进度与未收伏笔，逐项核对：

1. 当前阶段的完成标志是否全部达成，不能只看章节数量或单个高潮是否结束。
2. 最近十章的实际路线是否仍在推进当前阶段目标，还是已经形成持续偏移、替代路线或新的主要矛盾。
3. 核心读者承诺、人物目标、关键关系与硬约束是否仍与总纲一致。
4. 长期未推进的关键任务或伏笔是否已经影响阶段成立，需要调整近端规划或阶段表述。

审计没有发现偏移时，不改总纲、设定或未写章纲，但在收尾交付中明确写出“周期审计完成，无需修订”及依据。审计发现阶段完成、持续偏移或近端安排已不适用时，先输出“原计划 vs 实际走向 vs 后续影响”，等待用户确认后再修订，不能因到达十章节点便自动改写规划。

短篇模式默认跳过本步骤，除非用户明确要求调整尚未写的结局或方向。长篇触发后，读取当前十章章纲、总纲和相关设定，输出“原计划 vs 实际走向 vs 后续影响”。用户确认修订方案后再执行：

- **待定稿同步**：逐条说明定稿事实与现有设定、总纲或未写规划的差异。用户确认同步后，按影响范围执行 `webnovel-setting-sync`、更新总纲或修订未来章纲；用户暂缓时保留待定稿同步项，不静默改动全局文档。

- **近端调整**：只改未写的十章章纲或未写章节的章纲，不改已定稿事实。
- **当前阶段调整**：阶段尚未完成时，只更新 `3-大纲/3.1_全书结构总纲.md` 的当前阶段或入口承诺，不补写未来阶段。
- **阶段切换**：当前阶段已经达到完成标志时，先向用户说明实际收束结果，再执行 `webnovel-outline`。由该技能归档已完成阶段摘要，并根据最新定稿事实共创下一个当前阶段；本技能不能自行推演并写入下一阶段。
- **设定事实调整**：先执行 `webnovel-setting-sync`，处理角色、规则、核心机制或世界设定的跨文件一致性，再回写受影响的大纲。

未触发时跳过本步骤，不输出虚构的“复盘结果”。

## 步骤 5：按需文风进化

只有用户主动要求，或已积累多章用户认可的定稿正文并出现稳定模式时，才处理文风进化。单章临时写法、一次性情绪场景和偶然用词不能作为文风规则。

1. 选取多章定稿正文，提炼稳定的叙事视角与口吻、句段节奏、场景与信息释放、对白差异、情绪表达或本书特有避免项。
2. 先向用户展示“观察到的稳定特征”和对 `2-设定/2.4_文风.md` 的拟改动；用户确认后再更新。
3. 更新时优先合并或替换原有表述，不能只在原文后持续追加。每个维度保持两至三条高优先级倾向，不放示例句，不重复 `AGENTS.md` 的硬规则。
4. 只有跨多章反复稳定的特征才同步到对应维度；项目已有“进化记录”时，只追加简短的来源范围和变更摘要，不把记录扩写成第二份文风规则。

## 收尾交付

告知用户：本章已定稿、章纲是否已对齐、控制台实际更新项、本章所属十章章纲文件，以及是否触发规划或文风维护。之后等待用户要求下一章、调整设定或复查，不自动继续写作。

**下一步建议**：完成后，回到 `webnovel-zhengwen` 继续写下一章。全书完成时流程结束。
