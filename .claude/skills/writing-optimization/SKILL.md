# 正文润色

## 目标与边界

在不改变已确认的剧情事实、人物关系、视角、伏笔方向和设定边界的前提下，让正文更清楚、准确、顺畅并贴合本书文风。

本技能只调整现有正文的语言和场面呈现，不评判或重设剧情设计、人物目标、动机、关系与设定。

- 上位规则是项目根目录 `AGENTS.md`。
- 将本技能目录记为 `SKILL_DIR`，包含 `.agents/` 的项目根目录记为 `WORKSPACE_ROOT`。

## 固定输出

每次润色都从同一份原文独立生成简洁、调整、强化三份全文，完成比较并推荐一版。

### 三档核心差异

三档不是删多删少的渐变，而是减、调、强三种执行逻辑：

| 版本 | 核心方向 | 操作重心 |
| :--- | :--- | :--- |
| 简洁版 | 做减法 | 删除重复信息、弱修饰、拖慢节奏的解释和不必要反应；可以重排句子，但不得删掉必要剧情信息、情绪落点或伏笔。 |
| 调整版 | 做调整和补充 | 保留原文风格和剧情事实，允许补充必要信息和词语、替换弱词、拆并句子、调整分段与句序，让因果、判断链、情绪反应、冲突后果或段尾落点成立。 |
| 强化版 | 做结构性重写式强化 | 在事实边界内重排镜头、重建段落节奏，补足动作链、情绪链、判断链、环境反馈、冲突后果和段尾余味。不能停留在换词、补短句或轻微扩写，也不能靠堆辞藻和机械拉长制造强化感。 |

## 执行流程

### 1. 读取上下文

读取用户提供的段落或草稿。项目写作固定读取根目录 `AGENTS.md` 和 `2-设定/2.4_文风.md`；除此之外不预读任何章纲、设定、角色资料、关系记录、最近定稿或其他正文。

先以用户当前原文、`AGENTS.md` 和文风完成初步校验。只有具体润色改动需要确认既有剧情事实、能力边界、规则或前后连续性时，才读取能解决该问题的最少相关条目；不得为求完整而读取整份文件或无关材料。

### 2. 功能判断与专项路由

先判断原文承担的功能：推进、转场、冲突、打斗、伏笔、信息或情绪落点。只优化原文已有功能的呈现。

专项只辅助语言与场面呈现：

- 原文以打斗、追杀、逃生或动作交锋为主时，读取 `<WORKSPACE_ROOT>/.agents/skills/combat-writing/SKILL.md`，再按其指南读取最少的相关参考；只优化动作呈现，不改变胜负、能力或战术结果。
- 原文出现明显流水账，即事件依次平推、轻重不分，压迫、转折或结果落点不清时，读取 `<WORKSPACE_ROOT>/.agents/skills/conflict-design/SKILL.md`；只调整既有事件内的冲突节拍和落点，不新增剧情事件、两难、反转或章尾钩子。

主打斗段同时存在流水账问题时可以读取两个专项；其他情况不加载专项。专项不套用自己的交付格式，也不能改变原文事实或本技能的三档输出。

### 3. 具体策略

只执行原文实际命中的策略。没有问题的句段保留，不为证明润色过而制造改动。

#### 策略零：原文形态保留

先判断原文的叙事视角、句段密度、对话比例和作者手感。结果应像作者自己精修后的版本。简洁版和调整版优先保留原文形态；强化版可以重排镜头和段落，但必须守住剧情骨架、人物状态、视角与文风底色。

#### 策略一：主语与信息检查

检查读者第一遍能否说清谁做了什么、作用到谁或什么、造成什么结果。无主语动作会让读者回找执行者时补上主语；人物不知道的信息不得由叙述替他补全。

#### 策略二：动作压力与紧迫感检查

紧张场面先检查危险来源、压迫动作、不行动的损失、行动代价和局面变化。优先把动作和结果写准，不机械添加副词、感官、碎短句或新的冲突。

#### 策略三：状态转折检查

遇到外人离开、门关上、危险解除、身份切换、独处、胜负已分或进入安全区等节点，检查人物状态是否随局面变化。用选择、动作、说话方式、停顿或明显结果承接，不凭空新增心理动机。

#### 策略四：场景支撑与反差检查

检查空间、物件、光线、声音、温度和人物位置是否支撑行动、判断或情绪。存在真实反差时，可以用现场已有事物表现；没有来源或信息作用的气味、材质、五感和比喻直接删除。

#### 策略五：人物判断链检查

角色产生怀疑、警觉、害怕、愤怒、信任或其他判断时，检查读者能否看到依据。优先写成“可见事实或已知信息 -> 与常理、身份或前文冲突 -> 人物作出判断或选择”，不只给结论，也不重复解释已经写清的依据。

#### 策略六：情绪反应链检查

人物受压、松弛、羞恼、暴怒、恐惧或崩溃时，检查语言、动作和选择是否相互支撑。优先使用会改变交流或行动的反应；身体细节只留一两个能提供新信息的，不按固定顺序反复使用呼吸、眼神、脸色和手抖。

#### 策略七：伏笔与线索可读性检查

身份破绽、危险预警、关系变化、规则漏洞和后文钩子必须有具体承载物或可见事实。让读者能够察觉异常，但不新增答案、不提前揭谜，也不把留白误判成缺少解释。

#### 策略八：润句与润场区分

先判断问题位于哪一层：

1. **语言层**：主语、用词、句序、重复、解释腔、压缩短语和节奏。
2. **段落层**：动作因果、状态转折、场景支撑、判断链、反应链、冲突结算和段尾落点。

只改词句无法解决段落功能缺口时，调整版或强化版才进入段落层；简洁版不能为了收短删除必要的因果和落点。

#### 策略九：有效改动检查

润色不能等同于保守校对。有效改动包括词语更准、节奏更稳、落点更清、视角更稳、场面更足和链条更完整。

- 调整版应体现至少两类有效改动。
- 强化版应体现至少三类有效改动，且至少包含一类段落层或场面层改动。
- 原文不支持某项改动时保留，不为达到数量虚构内容。

### 4. 生成三版

不论原文长短，都在一次任务中从同一份原文独立生成简洁、调整、强化三份完整全文，不拆分、不暂存，也不在前一版上继续修改。不得因篇幅缩短三档、遗漏正文或改成单档交付。

### 5. 输出与自查

每版完成后、输出正式结果前逐项检查：

1. **档位方向与差异**：简洁版做减法，调整版做调整和必要补充，强化版做结构性强化；三版不能只是删多删少。
2. **强化达到段落层**：强化版必须形成镜头、节奏、动作链、判断链、反应链、环境反馈、冲突后果或段尾余味中的结构性补强。
3. **事实与规则**：三版不改变剧情事实、人物决定、关系结论、能力设定、关键道具、谜底或 `AGENTS.md` 及已调用专项的约束。

进入比较页前，整理原文功能、实际调用的专项、推荐版本和不超过三条理由。

## 静态网页预览页

完成三档润色后，使用文件写入能力将比较数据直接写入临时文件 `.temp\润色结果预览数据.json`，再在项目根目录执行固定脚本，生成或覆盖唯一的 `.temp\润色结果预览页.html`。不要手写 HTML、改样式或另建历史页。

```bash
python .agents/skills/writing-optimization/scripts/render_compare_html.py --input .temp\润色结果预览数据.json --output .temp\润色结果预览页.html
```

不要把原文或三档正文嵌入 Shell 命令、命令变量、here-string、`--json` 参数或标准输入管道；这些方式会受命令长度和引号转义限制。临时 JSON 的字段为 `source`、`function`、`versions`（简洁、调整、强化三项）和 `recommendation`；推荐从原文或三档中选择一版并说明理由，只允许使用 `<strong>` 和 `<br>`。渲染成功后删除临时 JSON，只保留预览页。

```json
{"title":"润色结果预览页","source":"原文","function":"剧情功能诊断","versions":[{"name":"简洁版","text":"正文"},{"name":"调整版","text":"正文"},{"name":"强化版","text":"正文"}],"recommendation":"推荐<strong>版本名</strong>。\n\n理由：<推荐理由>"}
```

脚本不可用时说明阻塞原因，不手写替代页面。

聊天输出默认只有三行：

1. `绝对路径入口：` 后接行内代码包裹的脚本返回绝对路径。
2. `Markdown 链接入口：` 后接 `[打开预览页](链接目标)`；目标将 Windows 路径中的 `\` 转为 `/`，并在盘符前加 `/`，不得使用相对路径或 `file:///`。
3. `如果已经打开过预览，可以在原预览页看到最新内容。`

不要在聊天里展示脚本执行过程或完整正文；只有用户明确要求时才在聊天中输出正文。HTML 中保留原文、剧情功能、三档正文和推荐理由。
