# 番茄正文 (Write)

## Usage
触发指令：`加载正文`、`写正文`、`写一章`、`番茄正文`、`审查正文`、`审查章节`

## 目标
将总纲/方向大纲转化为适合番茄移动端阅读的高质量正文；每写第 X 章前，先即时生成并确认第 X 章章纲，再按本章章纲分场景撰写，保证首屏抓力、单章推进、爽点兑现、章尾追读与去 AI 味，并更新角色状态、伏笔与进度。

## 去 AI 味总则

本技能在正文创作、完稿校准、章节审查三个阶段，都必须显式参考以下全局规则文件：

- `.agents/rules/anti-ai-expression.md`（禁令清单：不能写什么）
- `.agents/rules/anti-ai-rhythm.md`（韵律约束：段落/句式/换行）
- `.agents/rules/writing-optimization.md`（正向优化策略：改比删好）
- `.agents/rules/character-realism.md`（人物真实性）

执行要求：

- 这三份规则用于防止机械套话、无效抒情、过度解释、节奏匀速和人物失真；它们服务正文质量，不凌驾于番茄网文的留存、爽点和可读性之上。
- 番茄长篇正文的优先级为：首屏抓力 / 爽点清楚 / 剧情逻辑 / 人物真实 / 去明显 AI 味 / 局部表达新鲜度。若“去 AI 味”建议会削弱读者理解、爽点兑现或移动端节奏，应保留清楚有效的原表达。
- 不机械禁用单个常见词或网文化表达。只有当某类词句在同章高频重复、替代了具体动作、造成假抒情或机器感时，才视为问题；常见但位置合适、功能明确、节奏有效的句子可以保留。
- 审查时不要因为“网文化”“直给”“解释规则”本身扣分。番茄爽文允许直给和熟练句式；应扣分的是重复、虚假、空泛、机械、人物不像人、解释抢戏。
- 审查人物时按角色功能分层，不要求每章、每个角色都做到完整人物刻画。工具人、过场敌人和功能型角色只要动机可信、完成场景功能、不降智、不抢戏即可。
- 若 `1-边界/1.2_文风.md`、项目宪法 `.novelkit/constitution/MASTER.md` 与全局规则存在交叉，以“题材正确 + 去 AI 味 + 人物真实”三者兼容为目标执行。
- 若用户明确指定文风，以用户要求为最高优先级，但仍应尽量规避机械表达、套路句式和失真人物反应。

## 📌 阶段开始前：回顾预期

**【立即提示用户】**：
> 进入正文阶段前，建议先打开 `预期.md`，回顾你的初始执念、番茄读者情绪与题材边界，确认正文写作方向与预期一致，再开始动笔。

## 步骤 1：写作前置检查 (Pre-Writing Check)

- **目标**：确保动笔前的所有信息输入是完整且准确的。
- **动作**：在撰写 **每一章** 正文前，必须检查并整合以下关键信息：
  1. **风格约束**：
     - `1-边界/1.2_文风.md`（基础文风 — 路径已注册于 `SOLOENT.md §5`）
     - `1-边界/1.5_微观节奏拆解.md`（如有：导演手册，场景调度/镜头语言）
  2. **红线底线**：项目宪法 `.novelkit/constitution/MASTER.md`（路径已注册于 `SOLOENT.md §6`）
  3. **去 AI 味底线与正向策略**：
     - `.agents/rules/anti-ai-expression.md`（禁令清单）
     - `.agents/rules/anti-ai-rhythm.md`（韵律约束）
     - `.agents/rules/writing-optimization.md`（正向优化策略）
     - `.agents/rules/character-realism.md`（人物真实性）
  4. **设定支持**：
     - 查阅 `SOLOENT.md §2`（世界观索引）和 `§3`（角色索引），按指针读取对应文件
     - `2-设定/2.3_金手指设定.md`（如有：若本章涉及使用）
  5. **当前状态**：`.novelkit/memory/character_state.md`（路径注册于 `SOLOENT.md §7.2`，确保获取最新状态）
  6. **伏笔与进度**：
     - `.novelkit/memory/foreshadowing.md`（路径注册于 `SOLOENT.md §7.4`）
     - `SOLOENT.md §8`（当前阶段与待办）
  7. **大纲与本章指令**：
     - 读取 `3-大纲/3.1_全书结构总纲.md`
     - 读取 `3-大纲/XX阶段_方向锚点.md`（如有）
     - 检查 `3-大纲/XX卷_章纲.md` 中是否已有第 X 章章纲
     - 若没有第 X 章章纲，必须先执行 **步骤 1.1：即时章纲生成**，确认后再继续
  8. **上下文参考**：读取 **上一章正文**（确保剧情连贯、语气承接）。
- **目的**：防止逻辑BUG、风格漂移和设定遗忘。

## 步骤 1.1：即时章纲生成 (Just-in-time Chapter Beats)

- **目标**：写哪一章，只补哪一章章纲，让本章施工图贴合上一章真实正文和最新创作判断，避免被开书阶段的早期细纲绑死。
- **触发条件**：撰写第 X 章正文前，若本章章纲不存在、用户提出新想法、上一章正文已经偏离旧章纲，或旧章纲不符合当前要求，必须执行本步骤。
- **输入**：
  1. `3-大纲/3.1_全书结构总纲.md`
  2. `3-大纲/XX阶段_方向锚点.md`（如有）
  3. 设定、角色、金手指、宪法、文风、微观节奏拆解
  4. 上一章正文、角色状态、伏笔进度、SOLOENT 当前阶段与待办
  5. 用户对本章的最新意图或限制（如有）
- **动作**：
  - 只生成第 X 章章纲，不批量生成后续章节章纲。
  - 章纲必须包含：章节梗概、本章番茄目标、前 200 字首屏抓力、场景拆解、爽点/虐点、伏笔与线索、结尾钩子、不可写偏的边界。
  - 若存在旧第 X 章章纲但已不符合上一章正文或用户新要求，应先说明差异，再给出修订版。
  - 将确认后的本章章纲追加/更新到 `3-大纲/XX卷_章纲.md`；文件不存在则创建。
- **强制暂停**：生成或修订本章章纲后，**禁止直接写正文**，等待用户确认：
  > 「第 X 章即时章纲已生成/修订，请确认本章方向、场景和结尾钩子无误后回复『继续』进入正文。」

## 步骤 1.5：镜头稿转换 (Scene Camera Pass)

- **目标**：先把章纲任务转成可见、可听、可感的现场，再进入正文，避免把大纲说明直接翻译成小说段落。
- **动作**：在撰写 **每一章** 正文前，必须按已确认的本章即时章纲场景顺序生成内部镜头稿。镜头稿不必展示给用户，除非用户要求，但写正文时必须以它为底稿。
- **每个场景必须包含**：
  1. **现场画面**：读者第一眼看见/听见/闻到/摸到什么；优先使用门、灯、血迹、饭食、药膏、衣物、腕牌、脚步、沉默等可感细节。
  2. **人物动作**：主角、视角人物和关键配角分别做了什么；动作必须贴合伤势、身份、恐惧、利益和当下目标。
  3. **情绪压力**：本场要让读者感到紧张、憋屈、爽、寒意、期待、后怕或别的具体情绪。
  4. **信息藏法**：章纲中的设定、伏笔、关系变化、待遇变化、危机升级，分别由哪个物件、动作、称呼、对话或旁人反应承载。
  5. **禁写说明**：列出本场最容易写成说明文的句子，并禁止照写。
- **镜头稿原则**：
  - 先找“能被看见的证据”，再写人物判断；先写手、眼、声音、停顿、物件变化，再写结论。
  - 不把“这个场景的作用”“设定的意义”“人物为什么害怕”直接告诉读者；让读者从现场反应里自己明白。
  - 必要规则可以直给，但必须短、准、服务爽点；不能连续用解释句替代戏剧行动。

## 步骤 2：正文撰写与AI辅助 (Writing with AI)

- **输入**：步骤1中整合的全部信息。
- **动作**：
  - **从镜头稿落正文**：必须先完成步骤 1.5，再从镜头稿中的现场画面、人物动作和信息藏法展开正文；禁止把章纲条目直接扩写成说明文。
  - **分场景生成**：严格按照本章即时章纲拆解的场景顺序，**逐个场景**进行撰写。
  - **字数锚定**：建议单章总字数在 **2300-4000 字** 之间（用户可根据平台要求自行指定范围）。
  - **首屏抓力**：开章前 200 字内给出明确问题、冲突、反差或情绪压力，避免慢铺设定。
  - **文风执行**：严格执行 `1-边界/1.2_文风.md` 的要求，用动作、对话、细节承载信息，减少大段说明。
  - **低解释正文**：优先用物件变化、待遇变化、权限变化、旁人避让、短对话、身体反应和现场后果承载信息；少用“他明白了”“这意味着”“真正可怕的是”等总结性旁白。
  - **爽点兑现**：本章至少完成一次明确推进，避免只铺垫不回报；憋屈可存在，但必须给读者下一步反击预期。
  - **反 AI 表达执行**：逐场景检查是否出现黑名单词、陈词滥调、总结性旁白、机械化动作链和均匀段落节奏；发现后当场改写，不留到最后集中修补。
  - **人物真实性执行**：人物说话和反应必须贴合身份、处境、利益与情绪，不允许只为推进剧情而统一腔调。
  - **结尾钩子**：确保章节末尾包含本章即时章纲中设计的悬念、危机、反转、新收益或关系爆点钩子。
  - **创作范围**：严格遵守本章即时章纲剧情范围。若写作中出现更优剧情方向，先暂停并询问是否修订本章章纲，确认后再继续正文。
  - **草稿前低解释自查**：保存草稿前，删改以下类型句子：
    - “这意味着……”
    - “他意识到……”
    - “真正可怕的是……”
    - “这不是……而是……”
    - “这说明……”
    - 直接解释剧情功能、主题、设定意义、人物恐惧来源的句子
  - **节奏自查**：短句可以使用，但不能全章都像判定句；必须有动作推进、对话停顿、感官细节和长短句变化。
- **输出产物**：`4-正文/第X章_草稿.md`。
- **【强制暂停】** 草稿生成后，**禁止自动进入步骤 3，等待用户明确回复**：
  > 「第 X 章草稿已生成（`4-正文/第X章_草稿.md`），请阅读确认后回复『继续』进行完稿校准。」
- 收到用户『继续』（或等效肯定）后，方可进入步骤 3。

## 步骤 3：完稿后校准 (Post-Writing Calibration)

- **动作**：
  1. **严格审查（八项检查）**：
     - **字数检查**：确保单章字数在2300-4000字之间（使用 `scripts/count_chinese_words.py` 统计），可以超出 4000 字，但不能低于 2300 字。
       - 评分时，字数在用户指定范围或默认 2300-4000 字范围内即视为满分；不要因为刚过最低线而扣分。若节奏拖沓或信息不足，应在留存、节奏或大纲符合度中评估。
     - **大纲符合性检查**：对照 章纲，检查本章核心事件、场景和结尾钩子是否完全按计划执行。
     - **番茄留存检查**：检查首屏是否抓人、本章是否有推进/爽点、章尾是否能带动下一章追读。
     - **文风检查**：对照 `1-边界/1.2_文风.md` 和项目宪法，检查是否存在AI写作痕迹。
     - **反 AI 表达检查**：逐项对照 `.agents/rules/anti-ai-expression.md`，检查是否存在黑名单词、套路句、无意义总结、修饰语堆砌。
     - **反 AI 韵律检查**：逐项对照 `.agents/rules/anti-ai-rhythm.md`，检查动作链是否机械、句式呼吸是否单一、段落节奏是否过匀。
     - **人物真实性检查**：对照 `.agents/rules/character-realism.md`，检查人物动机、对话、反应和行为是否真实且前后一致。
     - **接续检查**：对照上一章，检查剧情、时间、地点是否无缝接续。
     - **金手指合规性**：（若涉及）检查表现形式是否符合 相关金手指设定文件。
     - **低解释正文检查**：检查章纲信息是否由画面、动作、对话、物件和后果承载；若关键段落主要靠解释句推进，必须重写该段。
     - **说明句清理**：重点搜索并删改“这意味着 / 他意识到 / 真正可怕的是 / 这不是……而是…… / 这说明”等句式，除非该句是角色台词且符合人物状态。
  2. **显式输出【完稿自检卡】**：至少包含“字数 / 大纲符合 / 番茄留存 / 文风 / 反 AI 表达 / 反 AI 韵律 / 人物真实性 / 接续与设定”八栏结果。如果有一项不通过，必须重新创作，或者进行相应修订。
- **目的**：保持设定与正文的实时同步，确保单章质量达标。

## 步骤 4：章节审查（可选，但强烈建议用于关键章） (Chapter Review)

- **适用时机**：
  - 用户明确说“审查这一章”“看看这一章有没有 AI 味”“帮我精修这一章”
  - 关键章节，如开篇章、爆点章、反转章、收束章
  - 连续多章写完后，需要联审节奏、逻辑和风格统一性
- **动作**：
  1. 若存在 `docs/提示词-05-章节审查.md`，优先读取并遵循。
  2. 审查前必须加载：
     - `1-边界/1.2_文风.md`
     - `.novelkit/constitution/MASTER.md`
     - `.agents/rules/anti-ai-expression.md`
     - `.agents/rules/anti-ai-rhythm.md`
     - `.agents/rules/character-realism.md`
     - 本章章纲、上一章正文、待审正文
  3. 输出完整审查报告，至少覆盖：
     - 字数
     - 大纲符合度
     - 文风与题材一致性
     - 反 AI 表达
     - 反 AI 韵律
     - 人物真实性
     - 上下文接续
     - 首屏抓力
     - 爽点兑现
     - 结尾钩子有效性
  4. 对每个问题给出具体定位和可执行修改建议；必要时给出“建议重写本章”结论。
- **输出产物**：`审查/第X章-审查.md`（若用户要求落盘时）。

## 步骤 5：伏笔、状态与进度更新 (Foreshadowing, State & Progress Update)

⚠️ **CRITICAL STEP**: 此步骤决定了记忆的连续性。绝对不可跳过！

- **状态回写**：根据本章发生的实际剧情，**立即更新**以下文件：
  - `.novelkit/memory/character_state.md` → 角色状态变化（升级/道具/关系）
  - `.novelkit/memory/foreshadowing.md` → 新增或收回的伏笔条目
  - `SOLOENT.md §8` → 将本章标记为已完成，填入下章任务
- **执行确认**：必须明确输出"已更新 memory 文件"的反馈，方可结束本章创作循环。

## 🚫 循环阻断 (Loop Break)

- **动作**：在完成上述所有步骤（1-5）后，**必须暂停**。
- **状态检查与分支选择**：
   1. **正常推进**：
      - 检查总纲/阶段锚点、角色状态和伏笔进度。
      - **继续下一章**：询问 "第 X 章完成。是否继续生成第 X+1 章即时章纲？"
      - **阶段已完**：提示 "当前阶段完成。是否补充下一阶段方向锚点，或直接进入下一章即时章纲？"
   2. **剧情偏离/修订**：
      - 若本章剧情与章纲差异较大，或用户表示"有了新想法"。
      - **引导**："检测到/你提到剧情有变化，是否先修订下一章即时章纲？若影响总纲，再进行【动态修订】。"
- **等待**：等待用户明确指令，严禁自动开始下一章。
