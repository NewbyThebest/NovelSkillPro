# 番茄正文

## 目标与路径

每次只完成一章：读取当前事实，确定本章功能，整理章纲，写出草稿并先自查修正，再交付用户决定审查、润色、修改或定稿。

- 将本技能目录记为 `SKILL_DIR`，包含 `.agents/` 的项目根目录记为 `WORKSPACE_ROOT`。
- 小说文件从 `WORKSPACE_ROOT` 读取和写入。调用字数脚本时，使用 `<SKILL_DIR>/scripts/count_chinese_words.py` 的实际绝对路径，工作目录保持 `WORKSPACE_ROOT`。
- 文本表达、排版和人物真实性始终遵循项目根目录的 `AGENTS.md`；本技能不重复其中的语言规则。

## 短篇模式覆盖

仅当用户明确创作短故事，或由 `webnovel-short-story` 调用并传入 `SHORT_STORY_MODE` 时使用：

- 用 `3-大纲/短篇方向大纲.md` 和 `3-大纲/短篇章纲.md` 代替长篇总纲、批次锚点和批次章纲文件。
- 跳过步骤 1；步骤 2 的本章方向从短篇方向大纲读取，章纲写入 `短篇章纲.md`。
- 默认单章 800-1500 字，用户指定范围优先；第一章将已确认的 `3-大纲/短篇导语.md` 放在正文开头。
- 步骤 3、步骤 4和草稿交付分支仍然执行，事实边界和定稿流程不变。

## 步骤 0：写作前置检查

1. 确保 `4-正文/` 存在，只创建缺失目录。
2. 长篇模式读取 `3-大纲/3.1_全书结构总纲.md`；短篇模式读取 `3-大纲/短篇方向大纲.md`。两种模式均读取 `2-设定/2.2_新书设定案.md`、`2-设定/2.4_主要角色设定表.md`、`2-设定/2.5_文风.md`；本章涉及核心机制时，再读取 `2-设定/2.3_金手指设定.md`。
3. 首章读取预期、设定和当前模式方向文件中的开篇承诺；第 2 章起读取上一章**定稿**、`控制台/角色状态.md`、`控制台/伏笔追踪.md`、`控制台/进度.md`。控制台文件在第一章定稿前不存在是正常情况。
4. 长篇模式确定本章所属批次文件：`3-大纲/第[卷名或序号]_章纲_[起始章]-[结束章]章.md`。短篇模式使用 `3-大纲/短篇章纲.md`。当前模式的章纲不存在时，长篇依次执行步骤 1 和步骤 2；短篇跳过步骤 1，直接执行步骤 2。
5. 已有本章 `_草稿.md` 或 `_定稿.md` 时，先确认用户是要修改已有文本还是新写；不得覆盖已有稿件。
6. 用户的新想法若改变世界规则、角色事实、核心机制或已确认大纲，先说明影响。未确认前不把它写进正文；需要变更时先执行 `webnovel-setting-sync`，再回到本章。

缺少会决定本章事实的材料时只问一次。不要为了补材料自行编造设定、角色关系、道具机制或前文事件。

## 步骤 1：批次锚点（仅长篇）

批次锚点是总纲与单章章纲之间的短期方向，不是固定十章模板。进入新的小剧情单元时创建，通常覆盖约 5-10 章；实际长度由阶段目标、事件完整度和用户计划决定。

短篇模式跳过本步骤。长篇模式读取总纲中当前阶段、上一批章纲末尾和控制台状态，写入本批次文件开头：

| 维度 | 内容 |
| :--- | :--- |
| 本批目标 | 本批结束后局面必须发生的变化 |
| 关键事件链 | 以因果顺序列出主要事件 |
| 核心压力或问题 | 这一批持续困住角色的矛盾 |
| 关系与角色推进 | 哪些关系、立场或独立利益会变化 |
| 核心机制（按需） | 机制的使用、限制、揭示或不涉及 |
| 伏笔与信息 | 应埋下、推进或回收的线索 |
| 读者回报 | 成长、破局、关系、情绪、悬念、获得或其他实际回报 |
| 节奏安排 | 哪些章节蓄势、推进、兑现或收束，不强制每章同强度 |

向用户确认批次方向后再进入本批第一章。用户已明确要求按已确认方向直接写时，可将确认与第一章章纲确认合并，不重复阻塞。

## 步骤 2：即时章纲

### 1. 先确定本章功能

判断本章主要属于开篇、承接、蓄势、兑现、关系/信息推进、转场或收束。功能决定开头强度、读者回报和结尾状态：

- 全书第一章前 200 字必须出现强压力、异常、冲突、反差或悬念。
- 承接章优先接住上一章留下的问题；常规、关系或转场章快速进入正在发生的事，不用背景、回忆或复述开场。
- 蓄势章可以不当场给大回报，但主角必须有观察、选择、准备或小幅推进，并让读者看到后续结算预期。
- 兑现章要让局面真实改变；收束章可以完成承诺、留下余波，不强行制造新钩子。

### 2. 取得本章思路

1. 先问用户是否已有本章思路。已有时以其为原料，不暗中改剧情。
2. 用户没有时，从长篇当前批次目标或短篇方向大纲、上一章结尾、未收伏笔、角色目标与状态、关系张力、外部压力和可用规则中，选择最相关的因素，提出 2-3 个不同方向。每个方向说明“本章推进什么、给读者什么、为什么现在发生”。
3. 用户选定方向后，可补充自己的展开。用户需要时再把方向扩成一段本章简介；不需要时直接进入建议。

### 3. 按需加载专项

- 本章核心是场景冲突、两难、反转或兑现时，读取 `conflict-design`。
- 本章核心是角色选择、关系拉扯或人物易写平时，读取 `character-vitality`。
- 本章存在战斗、追杀、对抗或斗法时，读取 `combat-writing`。

专项只补强命中的方面，不能覆盖已确认的本章方向。已在本次上下文读取过的专项直接沿用，不必重复读取。

### 4. 提出必要建议并整理

阅读用户原料后，只在确有价值时提出 0-3 条建议，说明改什么、为什么更好。用户采纳后才并入章纲；不得借建议私自新增设定或替换关键情节。

将确认内容写入长篇批次文件或短篇 `3-大纲/短篇章纲.md`。使用以下结构，允许“无”或“不适用”，不要为了填模板硬塞爽点、伏笔或钩子：

```markdown
### 第 X 章 · [标题]

**本章功能与目标**
[本章推进什么、读者为什么继续读]

**剧情骨架**
[按发生顺序写关键事件、选择和结果；不拆成正文镜头]

**读者回报或压力**
[本章兑现什么，或为后续结算积累什么]

**伏笔与线索**
[埋下、推进或回收的内容]

**结尾状态**
[钩子 / 结算 / 情绪余波；说明下一章为何值得读或本段为何收口]

**不可写偏的边界**
[本章不能改变的事实与限制]
```

旧章纲不再适用时，先说明与最新正文或用户要求的差异，再修订。通常等待用户确认章纲后写正文；用户明确授权直接写时，可以在说明章纲摘要后继续。

## 步骤 3：撰写草稿

1. 用场景、动作、对话和可见结果把章纲写成正文，不逐句翻译章纲或用解释填充。
2. 按章节功能控制开头与结尾，承接上一章已知事实和状态，不复用旧章的比喻、意象或套话。
3. 长篇默认参考 2000-4000 字，短篇模式默认参考 800-1500 字；用户指定范围时优先。字数服务于完整场景和有效推进，不靠补背景、重复情绪或无效反应凑数。
4. 依照 `2-设定/2.5_文风.md` 和最近的定稿正文保持本书手感；首章没有定稿可参考时，以种子文风和设定为准。
5. 写入 `4-正文/第X章_草稿.md` 后不立即交付，进入步骤 4。

## 步骤 4：完稿校准

草稿完成后先自查并直接修正可在不改变已确认剧情下解决的问题。

1. **字数**：在 `WORKSPACE_ROOT` 下执行：

   ```bash
   python "<SKILL_DIR>/scripts/count_chinese_words.py" X "<WORKSPACE_ROOT>/4-正文" --min-chars <参考下限> --max-chars <参考上限>
   ```

   将占位符替换为实际绝对路径、章节号和当前模式的参考范围：长篇默认 `2000` 至 `4000`，短篇默认 `800` 至 `1500`，用户指定范围优先。脚本只报告参考区间，不给正文判定“达标”或“不达标”；短于建议范围时，先检查是否缺少必要推进、结果或场景，不为达标硬补文字。
2. **章纲与阶段**：关键事件、章节功能、读者回报或蓄势预期、结尾状态是否按章纲落地，并仍对齐当前批次和总纲。
3. **连续性**：核对时间、地点、在场人物、伤势/资源、认知边界、因果、角色利益与前一章是否一致。
4. **核心机制**：本章涉及时，核对规则、限制、已知效果和暴露风险，不给角色临场新增能力。
5. **文风与可读性**：对照种子文风、最近定稿和 `AGENTS.md`，修正解释腔、重复、主语不清、动作无结果、人物反应失真和不合场景的细节。

输出自检卡，只报告实际检查与实际修正：

```markdown
## 完稿自检卡 · 第 X 章

| 检查项 | 结果 | 说明 |
| :--- | :--- | :--- |
| 字数 | XX 字 | 与用户要求或建议范围的关系 |
| 章纲与阶段 | 通过 / 已修正 | 本章功能、推进、回报或蓄势、结尾状态 |
| 连续性 | 通过 / 已修正 | 时间、空间、认知、因果、状态与前章接续 |
| 核心机制 | 通过 / 已修正 / 不涉及 | 规则与边界是否一致 |
| 文风与可读性 | 通过 / 已修正 | 对照文风和全局规则的实际问题 |

**本次修正**：[逐条记录；无则写“无”]
```

## 交付与后续分支

交付已校准的草稿和自检卡，等待用户选择下一步：

- 要全面质检本章时，执行 `webnovel-review`。
- 要精修表达、节奏或细节时，执行 `writing-optimization`。
- 用户自行修改时，等待其完成，不根据旧草稿提前回写状态。
- 用户确认本章定稿时，执行 `webnovel-dinggao`，以最新草稿或用户修改版为准完成转正、章纲回写和控制台更新。

不要把审查、润色或定稿当作草稿交付后的自动步骤。

**下一步建议**：用户确认本章定稿后，执行 `webnovel-dinggao` 完成定稿回写。之后回到 `webnovel-zhengwen` 写下一章。
