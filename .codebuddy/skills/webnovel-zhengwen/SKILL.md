# 番茄正文

## 目标与路径

每次只完成一章：读取当前事实，确定本章功能，整理章纲，写出草稿并先自查修正，再交付用户决定审查、润色、修改或定稿。

- 将本技能目录记为 `SKILL_DIR`，包含 `.agents/` 的项目根目录记为 `WORKSPACE_ROOT`。
- 小说文件从 `WORKSPACE_ROOT` 读取和写入。调用字数或句号换行检查脚本时，使用 `<SKILL_DIR>/scripts/` 下对应脚本的实际绝对路径，工作目录保持 `WORKSPACE_ROOT`。
- 文本表达、排版和人物真实性始终遵循项目根目录的 `AGENTS.md`；本技能不重复其中的语言规则。

## 短篇模式覆盖

仅当用户明确创作短故事，或由 `webnovel-short-story` 调用并传入 `SHORT_STORY_MODE` 时使用：

- 用 `3-大纲/短篇方向大纲.md` 和 `3-大纲/短篇章纲.md` 代替长篇总纲、批次锚点和批次章纲文件。
- 跳过步骤 1；步骤 2 的本章方向从短篇方向大纲读取，章纲写入 `短篇章纲.md`。
- 默认单章 800-1500 字，用户指定范围优先；第一章将已确认的 `3-大纲/短篇导语.md` 放在正文开头。
- 步骤 3、步骤 4和草稿交付分支仍然执行，事实边界和定稿流程不变。

## 步骤 0：写作前置检查

1. 确保 `4-正文/` 存在，只创建缺失目录。
2. 长篇模式读取 `3-大纲/3.1_全书结构总纲.md`；短篇模式读取 `3-大纲/短篇方向大纲.md`。两种模式均读取 `2-设定/2.1_新书设定案.md`、`2-设定/2.3_主要角色设定表.md`、`2-设定/2.4_文风.md`；本章涉及已确认核心关系时，读取 `2-设定/2.3.2_核心关系.md` 的相关条目；本章出现、复用或结算阶段角色时，读取 `2-设定/2.3.1_阶段与功能角色记录.md` 的相关条目；本章涉及核心机制时，再读取 `2-设定/2.2_金手指设定.md`。
3. 首章读取预期、设定和当前模式方向文件中的开篇承诺；第 2 章起读取上一章**定稿**、`控制台/角色状态.md`、`控制台/伏笔追踪.md`、`控制台/进度.md`。控制台文件在第一章定稿前不存在是正常情况。
4. 长篇模式确定本章所属批次文件。正在创作、结束章数未知的批次使用 `3-大纲/第[批次序号]批_章纲_[起始章]-x章.md`；已经收束的批次将 `x` 替换为实际结束章号。短篇模式使用 `3-大纲/短篇章纲.md`。当前模式的章纲不存在时，长篇依次执行步骤 1 和步骤 2；短篇跳过步骤 1，直接执行步骤 2。
5. 已有本章 `第X章_草稿_[标题].md` 或 `第X章_定稿_[标题].md` 时，先确认用户是要修改已有文本还是新写；不得覆盖已有稿件。同一章的标题在定稿前发生变化时，重命名现有草稿以反映最新标题，不另建一份重复草稿。
6. 用户确认的当前章纲或创作原料与现有设定、总纲或未写规划冲突时，定稿前不修改这些全局文档，也不提前执行 `webnovel-setting-sync`。写正文时按“用户最新确认的创作原料 > 当前章纲 > 现有设定与旧大纲”的顺序执行，并将实际差异写入本章“待定稿同步项”。只有冲突涉及已定稿正文或控制台中的既成事实时才停止并说明影响；用户明确确认追溯修改后再继续。

缺少会决定本章事实的材料时只问一次。不要为了补材料自行编造设定、角色关系、道具机制或前文事件。

## 步骤 1：批次锚点（仅长篇）

批次锚点只负责记录起点和终点，不规划中间过程。进入新的小剧情单元时创建，不预设包含多少章；达到本批目标的定稿章节就是本批结束章。

短篇模式跳过本步骤。长篇模式读取总纲中的当前阶段、上一批最后一章定稿和控制台状态。第一批尚无定稿正文时，使用已确认的开篇初始状态。创建 `第[批次序号]批_章纲_[起始章]-x章.md`，并只写入以下两项：

| 维度 | 内容 |
| :--- | :--- |
| 当前起点 | 创建本批时已经发生并确认的事实，不设计新剧情 |
| 本批目标 | 本批结束后局面必须发生的变化 |

- **当前起点**只引用定稿正文、控制台或已确认开篇状态中的事实，不写预测、可能性和下一步安排。
- **本批目标**只写结束时能够观察到的局面变化，不指定通过什么事件、角色、能力、线索或冲突达成。
- 批次锚点不得添加关键事件链、关系推进、核心机制安排、伏笔、读者回报、节奏计划或预计结束章数。这些内容在每章开始时根据最新定稿事实即时决定。
- 向用户确认当前起点和本批目标后再进入本批第一章。用户已明确要求按已确认方向直接写时，可将确认与第一章章纲确认合并，不重复阻塞。
- 每章定稿后由 `webnovel-dinggao` 判断本批目标是否已经达成。未达成时保留文件名中的 `x`；达成时将 `x` 改为该章章号。下一批从该章的下一章开始，重新创建新的 `起始章-x章` 文件。

## 步骤 2：即时章纲

### 1. 先确定本章功能

判断本章主要属于开篇、承接、蓄势、兑现、关系/信息推进、转场或收束。功能决定开头强度、读者回报和结尾状态：

- 全书第一章前 200 字必须出现强压力、异常、冲突、反差或悬念。
- 承接章优先接住上一章留下的问题；常规、关系或转场章快速进入正在发生的事，不用背景、回忆或复述开场。
- 蓄势章可以不当场给大回报，但主角必须有观察、选择、准备或小幅推进，并让读者看到后续结算预期。
- 兑现章要让局面真实改变；收束章可以完成承诺、留下余波，不强行制造新钩子。

### 2. 取得本章思路

1. 先问用户是否已有本章思路。已有时以其为原料，不暗中改剧情。
2. 用户没有时，从长篇当前批次目标或短篇方向大纲、上一章结尾、未收伏笔、角色目标与状态、关系张力、外部压力和可用规则中，选择最相关的因素，提出 2-3 个不同方向。每个方向说明“本章推进什么、给读者什么、为什么现在发生”。
3. 用户选定方向后，可补充自己的展开。用户需要时再把方向扩成一段本章简介；不需要时直接进入建议。

### 3. 按需加载专项

- 本章核心是场景冲突、两难、反转或兑现时，读取 `conflict-design`。
- 本章核心是角色选择、关系拉扯或人物易写平时，读取 `character-vitality`。
- 本章存在战斗、追杀、对抗或斗法时，读取 `combat-writing`。

专项只补强命中的方面，不能覆盖已确认的本章方向。已在本次上下文读取过的专项直接沿用，不必重复读取。

### 4. 提出必要建议并整理

阅读用户原料后，只在确有价值时提出 0-3 条建议，说明改什么、为什么更好。用户采纳后才并入章纲；不得借建议私自新增设定或替换关键情节。

用户提供或修改本章材料后，按以下顺序即时更新当前章纲，不等章节定稿：

1. 将用户最新版详细材料写入“创作原料（定稿前临时保留）”。只整理标点、明显错字和阅读格式，不把具体事件、对白、动作、场景细节或表达意图压缩丢失。
2. 立即按最新版创作原料修正本章“剧情骨架”及其他受影响字段，不让当前章纲与创作原料保留两个互相冲突的版本。
3. 如当前章节要求与现有设定、总纲、批次锚点或未写规划冲突，只提醒具体差异。用户认可当前章节要求后，立即按其要求完成本章章纲更新，不重复阻塞，并将差异记入“待定稿同步项”。

创作原料不是已发生事实，也不是最终正文。当前章节讨论期间，以“用户最新确认的创作原料 > 当前章纲 > 现有设定、总纲、批次锚点与未写规划”的顺序处理。只有已定稿正文和控制台中的既成事实不能被静默覆盖；如需追溯修改，说明影响并取得用户确认。用户后续修改思路时，用最新确认版本替换对应内容，不累计互相冲突的旧版本。

章节定稿前，只允许更新当前章节的章纲条目、创作原料和草稿。不得因此修改设定、总纲、批次锚点、控制台、其他章节章纲或其他全局资料，也不提前执行 `webnovel-setting-sync`。定稿后由 `webnovel-dinggao` 根据最终正文处理需要同步的全局资料。

将确认内容写入长篇批次文件或短篇 `3-大纲/短篇章纲.md`。使用以下结构，允许“无”或“不适用”，不要为了填模板硬塞爽点、伏笔或钩子：

```markdown
### 第 X 章 · [标题]

**本章功能与目标**
[本章推进什么、读者为什么继续读]

**剧情骨架**
[按发生顺序写关键事件、选择和结果；不拆成正文镜头]

**读者回报或压力**
[本章兑现什么，或为后续结算积累什么]

**伏笔与线索**
[埋下、推进或回收的内容]

**结尾状态**
[钩子 / 结算 / 情绪余波；说明下一章为何值得读或本段为何收口]

**不可写偏的边界**
[本章不能改变的事实与限制]

**待定稿同步项（定稿前临时保留）**
[当前章纲或创作原料与现有设定、总纲、批次锚点或未写规划的差异；没有时省略本节]

**创作原料（定稿前临时保留）**
> 本节不是已发生事实，也不是最终正文。写作时必须参考；与剧情骨架冲突时以用户最新确认的创作原料为准；与现有设定或旧大纲冲突时以当前章节要求为准，并记录待定稿同步项；只有已定稿正文和控制台既成事实不能被静默覆盖；章节定稿回写完成后删除。

[用户提供的最新详细思路；没有详细原料时省略本节]
```

旧章纲不再适用时，先说明与最新正文或用户要求的差异，再修订。通常等待用户确认章纲后写正文；用户明确授权直接写时，可以在说明章纲摘要后继续。

## 步骤 3：撰写草稿

1. 同时读取本章章纲与“创作原料（定稿前临时保留）”。两者冲突时以用户最新确认的创作原料为准，并先修正剧情骨架。当前章节要求与现有设定、总纲或未写规划冲突时，按当前章节要求写作并记录“待定稿同步项”，不提前修改全局文档；冲突涉及已定稿正文或控制台既成事实时，必须先说明影响。用场景、动作、对话和可见结果把已确认内容写成正文。创作原料里已经适合正文的对白、动作、场景细节和段落可以直接采用，不要求为了改写而改写；其余内容按正文需要展开，不能因为剧情骨架经过压缩便无故忽略。
2. 生成正文时直接按句排版。每写完一个以句号结束的完整句，立即换行；对话中的句号后有右引号时，写完右引号立即换行。下一句不得接在同一行。对话与说话人动作或神态使用逗号、冒号连接时，可以按 `AGENTS.md` 的例外同行。不要先写成连续段落，再把换行留给步骤 4。
3. 按章节功能控制开头与结尾，承接上一章已知事实和状态，不复用旧章的比喻、意象或套话。
4. 长篇默认参考 2000-4000 字，短篇模式默认参考 800-1500 字；用户指定范围时优先。字数服务于完整场景和有效推进，不靠补背景、重复情绪或无效反应凑数。
5. 依照 `2-设定/2.4_文风.md` 和最近的定稿正文保持本书手感；首章没有定稿可参考时，以种子文风和设定为准。文风文档表示全章的整体倾向，不要求每个段落逐条命中，不为迎合某条倾向扭曲场景需要、人物反应或自然节奏。
6. 从当前章纲取得章节标题，写入 `4-正文/第X章_草稿_[标题].md`。标题只写在文件名中，正文文件从第一句正文开始，不写 `# 第X章`、章节名或其他标题行。文件名中的半角禁用字符 `\ / : * ? " < > |` 使用合适的全角字符替换；没有自然替代时删除，并去掉标题末尾的空格或句点。写入后不立即交付，后续步骤始终使用该文件的实际绝对路径。

## 步骤 4：完稿校准

草稿完成后先执行句号换行自动校准，再自查并直接修正其他可在不改变已确认剧情下解决的问题。

1. **句号换行自动校准**：这是步骤 4 的固定第一项。在 `WORKSPACE_ROOT` 下执行：

   ```bash
   python "<SKILL_DIR>/scripts/check_sentence_linebreaks.py" --fix "<WORKSPACE_ROOT>/4-正文/第X章_草稿_[标题].md"
   ```

   将占位符替换为实际绝对路径和章节号。脚本直接修正句号或“句号加闭合引号”后仍有下一句正文的问题；拆句时保留一个空白行，中文引号内拆句时自动闭合并重新打开引号。固定运行一次后立即进入后续校准，不检查退出码，不根据脚本输出再次处理或重跑，也不因此中止、询问用户或单独交付结果。后续审查继续处理仍然存在的排版问题。
2. **字数**：在 `WORKSPACE_ROOT` 下执行：

   ```bash
   python "<SKILL_DIR>/scripts/count_chinese_words.py" X "<WORKSPACE_ROOT>/4-正文" --min-chars <参考下限> --max-chars <参考上限>
   ```

   将占位符替换为实际绝对路径、章节号和当前模式的参考范围：长篇默认 `2000` 至 `4000`，短篇默认 `800` 至 `1500`，用户指定范围优先。脚本只报告参考区间，不给正文判定“达标”或“不达标”；短于建议范围时，先检查是否缺少必要推进、结果或场景，不为达标硬补文字。
3. **章纲与阶段**：关键事件、章节功能、读者回报或蓄势预期、结尾状态是否按当前章纲落地；存在临时创作原料时，核对剧情骨架是否已与最新版原料一致，以及用户明确保留的内容是否被无故遗漏；与现有设定、总纲或未写规划仍有差异时，确认已经记入“待定稿同步项”，不得把这种待定差异当成草稿错误而擅自改回旧版本。
4. **连续性**：核对时间、地点、在场人物、伤势/资源、认知边界、因果、角色利益与前一章是否一致。
5. **核心机制**：本章涉及时，核对规则、限制、已知效果和暴露风险，不给角色临场新增能力。
6. **文风与可读性**：对照种子文风、最近定稿和 `AGENTS.md`，修正解释腔、重复、主语不清、动作无结果、人物反应失真和不合场景的细节；确认正文首行已经直接进入故事，没有重复文件名中的章号或标题。

输出自检卡，只报告实际检查与实际修正：

```markdown
## 完稿自检卡 · 第 X 章

| 检查项 | 结果 | 说明 |
| :--- | :--- | :--- |
| 字数 | XX 字 | 与用户要求或建议范围的关系 |
| 句号换行 | 已执行自动校准 | 已调用 `--fix`；结果不作为后续流程门槛 |
| 章纲与阶段 | 通过 / 已修正 | 本章功能、推进、回报或蓄势、结尾状态 |
| 连续性 | 通过 / 已修正 | 时间、空间、认知、因果、状态与前章接续 |
| 核心机制 | 通过 / 已修正 / 不涉及 | 规则与边界是否一致 |
| 文风与可读性 | 通过 / 已修正 | 对照文风和全局规则的实际问题 |

**本次修正**：[逐条记录；无则写“无”]
```

## 交付与后续分支

草稿交付的最终回复必须包含以下内容，缺少任一项都不算完成：

1. 给出已校准草稿的实际路径，并明确当前仍是草稿。
2. 输出完稿自检卡，只报告实际检查和修正。
3. 最后一句直接询问用户选择以下一项：
   - **全面质检**：执行 `webnovel-review`。
   - **精修正文**：执行 `writing-optimization`。
   - **作者自行修改**：等待用户提供最新版，不根据旧草稿提前回写状态。
   - **确认定稿**：执行 `webnovel-dinggao`，以最新版正文完成转正、章纲回写和控制台更新。

用户作出选择前保持草稿状态，不得自动执行质检、精修或定稿，也不得只提示“确认后定稿”。
