# 正文润色

## 目标与边界

在不改变已确认的剧情事实、人物关系、视角、伏笔方向和设定边界的前提下，让正文更清楚、准确、顺畅并贴合本书文风。

- 上位规则是项目根目录 `AGENTS.md`。
- 将本技能目录记为 `SKILL_DIR`，包含 `.agents/` 的项目根目录记为 `WORKSPACE_ROOT`。

## 固定输出

每次润色都从同一份原文独立生成简洁、调整、强化三份全文，完成比较并推荐一版。

### 三档核心差异

三档不是删多删少的渐变，而是减、调、强三种执行逻辑：

| 版本 | 核心方向 | 操作重心 |
| :--- | :--- | :--- |
| 简洁版 | 做减法 | 删除重复信息、弱修饰、拖慢节奏的解释和不必要反应；可以重排句子，但不得删掉必要剧情信息、情绪落点或伏笔。 |
| 调整版 | 做调整和补充 | 保留原文风格和剧情事实，允许补充必要信息和词语、替换弱词、拆并句子、调整分段与句序，让因果、判断链、情绪反应、冲突后果或段尾落点成立。 |
| 强化版 | 做结构性重写式强化 | 在事实边界内重排镜头、重建段落节奏，补足动作链、情绪链、判断链、环境反馈、冲突后果和段尾余味。不能停留在换词、补短句或轻微扩写，也不能靠堆辞藻和机械拉长制造强化感。 |

## 执行流程

### 1. 读取上下文

读取用户提供的段落或草稿。项目写作中，按需补读本章章纲、`2-设定/2.4_文风.md`、最近定稿和相关角色状态；涉及主要角色时读取 `2-设定/2.3_主要角色设定表.md` 的对应条目，涉及核心关系时读取 `2-设定/2.3.2_核心关系.md` 的对应条目，涉及阶段或功能角色时读取 `2-设定/2.3.1_阶段与功能角色记录.md` 的对应条目。

### 2. 功能判断与专项路由

先判断原文承担的功能：推进、转场、冲突、人物、关系、伏笔、信息或情绪回报。只补强实际承担的功能。

按内容读取所有命中的专项；已经在本次上下文完整读取过的直接沿用：

- 人物塑造、关系推进或角色反应：`<WORKSPACE_ROOT>/.agents/skills/character-vitality/SKILL.md`。
- 压迫、两难、反转、兑现或章尾追读：`<WORKSPACE_ROOT>/.agents/skills/conflict-design/SKILL.md`。
- 战斗、追杀、逃生或动作交锋：`<WORKSPACE_ROOT>/.agents/skills/combat-writing/SKILL.md`，再按其指南读取最少的相关参考。

多项同时命中时全部加载。专项只处理命中部分，不套用专项自己的交付格式，也不能改变原文事实、当前章纲或本技能的三档输出。推进、转场、伏笔、信息和普通情绪段没有命中专项时，直接使用下方通用策略。

需要新增剧情事件、改变胜负或人物决定、调整关系结论、补充未确认设定才能解决的问题，不得借润色直接改写。

### 3. 具体策略

只执行原文实际命中的策略。没有问题的句段保留，不为证明润色过而制造改动。

#### 策略零：原文形态保留

先判断原文的叙事视角、句段密度、对话比例和作者手感。结果应像作者自己精修后的版本。简洁版和调整版优先保留原文形态；强化版可以重排镜头和段落，但必须守住剧情骨架、人物状态、视角与文风底色。

#### 策略一：主语与信息检查

检查读者第一遍能否说清谁做了什么、作用到谁或什么、造成什么结果。无主语动作会让读者回找执行者时补上主语；人物不知道的信息不得由叙述替他补全。

#### 策略二：动作压力与紧迫感检查

紧张场面先检查危险来源、压迫动作、不行动的损失、行动代价和局面变化。优先把动作和结果写准，不机械添加副词、感官、碎短句或新的冲突。

#### 策略三：状态转折检查

遇到外人离开、门关上、危险解除、身份切换、独处、胜负已分或进入安全区等节点，检查人物状态是否随局面变化。用选择、动作、说话方式、停顿或明显结果承接，不凭空新增心理动机。

#### 策略四：场景支撑与反差检查

检查空间、物件、光线、声音、温度和人物位置是否支撑行动、判断或情绪。存在真实反差时，可以用现场已有事物表现；没有来源或信息作用的气味、材质、五感和比喻直接删除。

#### 策略五：人物判断链检查

角色产生怀疑、警觉、害怕、愤怒、信任或其他判断时，检查读者能否看到依据。优先写成“可见事实或已知信息 -> 与常理、身份或前文冲突 -> 人物作出判断或选择”，不只给结论，也不重复解释已经写清的依据。

#### 策略六：情绪反应链检查

人物受压、松弛、羞恼、暴怒、恐惧或崩溃时，检查语言、动作和选择是否相互支撑。优先使用会改变交流或行动的反应；身体细节只留一两个能提供新信息的，不按固定顺序反复使用呼吸、眼神、脸色和手抖。

#### 策略七：伏笔与线索可读性检查

身份破绽、危险预警、关系变化、规则漏洞和后文钩子必须有具体承载物或可见事实。让读者能够察觉异常，但不新增答案、不提前揭谜，也不把留白误判成缺少解释。

#### 策略八：润句与润场区分

先判断问题位于哪一层：

1. **语言层**：主语、用词、句序、重复、解释腔、压缩短语和节奏。
2. **段落层**：动作因果、状态转折、场景支撑、判断链、反应链、冲突结算和段尾落点。

只改词句无法解决段落功能缺口时，调整版或强化版才进入段落层；简洁版不能为了收短删除必要的因果和落点。

#### 策略九：有效改动检查

润色不能等同于保守校对。有效改动包括词语更准、节奏更稳、落点更清、视角更稳、场面更足和链条更完整。

- 调整版应体现至少两类有效改动。
- 强化版应体现至少三类有效改动，且至少包含一类段落层或场面层改动。
- 原文不支持某项改动时保留，不为达到数量虚构内容。

### 4. 生成三版

三版都从原始原文独立生成。每写完一版后重新读取原文，再写下一版，不在前一版上继续修改。

### 5. 输出与自查

每版完成后、输出正式结果前逐项检查：

1. **档位是否匹配**：简洁版应以删除提纯为主；调整版应以替换、调整和补充为主；强化版应以结构性重写式强化为主。方向不符时重写该版。
2. **三版是否真正不同**：三版必须体现三种操作方向；若只是删多删少，调整版和强化版中偏弱的一版必须重写。
3. **强化是否达到段落层**：强化版若只换词、补短句或轻微扩写，没有形成镜头、节奏、动作链、判断链、反应链、环境反馈、冲突后果或段尾余味的结构性补强，必须重写。
4. **事实边界是否一致**：强化版可以新增现场细节和承接动作，但不得新增会改变剧情走向的人物决定、关系结论、能力设定、关键道具、阴谋解释或后文谜底。
5. **专项与项目规则是否落实**：命中的人物、冲突或打斗专项已经执行，三版都符合 `AGENTS.md`。

进入比较页前，整理原文功能、实际调用的专项、三版全文、推荐版本和不超过三条理由。

## 静态网页预览页

每次完成三档润色后，必须在项目根目录执行固定脚本，把预览数据通过标准输入喂给脚本，生成或覆盖项目下 `.temp` 目录的 `润色结果预览页.html`（路径：`.temp\润色结果预览页.html`）；不要手写 HTML、不要改样式、不要生成历史文件。若当前工作目录不确定，先切到项目根目录，或使用脚本绝对路径。

```bash
python .agents/skills/writing-optimization/scripts/render_compare_html.py --stdin --output .temp\润色结果预览页.html
```

必须使用 `--stdin`，不要使用 `--json` 命令行参数传整段 JSON；若运行环境已经是 PowerShell，不要再用 `powershell -Command "..."` 或 `pwsh -Command "..."` 包裹整段正文/JSON/命令；直接在当前 PowerShell 上下文里用变量和管道执行。JSON 最小结构：`source`（原文）、`function`（剧情功能诊断）、`versions`（三个版本，每个含 `name` 和 `text`）。`recommendation` 格式：推荐版本标识（从原文/简洁版/调整版/强化版四个中选一个）+ 推荐理由。例：`推荐<strong>调整版</strong>。\n\n理由：<推荐理由>`。如需强调，只允许 `<strong>` 和 `<br>`。必须先在内存中组织 JSON，再直接写到脚本标准输入；不要创建临时 JSON 文件或其他一次性脚本/文件。

**单行执行兼容**（适用于 AI 工具等只能发送单行命令的环境）：
- 将 JSON 存入 PowerShell 单引号字符串变量，JSON 内部换行使用 `\n` 转义序列（Python `json.loads()` 会正确解析）
- 然后通过管道传给脚本：`$json = '{"key":"val\nval2"}' ; $json | python ...`
- 单引号字符串内唯一需转义的是单引号本身（双写 `''`）；JSON 字段值中避免使用英文单引号，或用全角 `「」` 替代

```json
{"title":"润色结果预览页","source":"原文","function":"剧情功能诊断","versions":[{"name":"简洁版","text":"正文"},{"name":"调整版","text":"正文"},{"name":"强化版","text":"正文"}],"recommendation":"推荐<strong>版本名</strong>。\n\n理由：<推荐理由>"}
```

脚本不可用时先说明原因，再手写同等结构预览页。

聊天输出默认只有三行：第一行写“绝对路径入口：”，后接 inline code 包裹的渲染脚本返回绝对路径；第二行写“Markdown 链接入口：”，后接 Markdown 链接 `[打开预览页](链接目标)`；第三行写 `如果已经打开过预览，可以在原预览页看到最新内容。`Markdown 链接目标将返回路径中的 `\` 转为 `/`，并在 Windows 盘符前加 `/`，例如 `F:\项目\.temp\润色结果预览页.html` 转为 `/F:/项目/.temp/润色结果预览页.html`。不得使用相对路径或 `file:///`。不要在聊天里展示"写入脚本/执行脚本/删除脚本/打开网页/执行成功/DONE"等过程信息。原文、三版完整正文、剧情功能、优化点和推荐理由都必须放入 HTML 预览页；页面以"原文 + 简洁版 + 调整版 + 强化版"并排对照。聊天窗口默认不输出正文代码块；只有用户明确要求聊天内正文时，才用代码块呈现，方便复制。
