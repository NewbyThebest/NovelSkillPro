# 章节优化策略引导

## 基础约束

- **不改变原意**：优化可以替换句式、调整措辞、改变表达方式，但不能扭曲或遗漏原文的核心信息。
- **按需补足**：三档都必须让文本达到"读者看得清、读得顺、功能成立"。
- **聊天输出只给预览入口**：默认只在聊天窗口输出一句兼容提示和两个预览入口：`哪个能点就点哪个，兼容不同编辑器。`；随后给 Markdown 文件链接和反引号包裹的文件名提示。Markdown 链接显示文本写 `润色结果预览页`，inline code 文件名写 `润色结果预览页.html`；不要显示完整路径或 `file:///` URL，不要重复粘贴剧情功能、三版要点、推荐理由或完整润色正文。支持本地 Markdown 文件链接的环境可点 Markdown 链接；CodeBuddy 等能识别 inline code 文件名的环境可点 inline code 文件名。只有用户明确要求"在聊天里也给正文/给推荐版正文/展开三版/解释理由"时，才在聊天中输出相应内容。
- **默认生成静态预览页**：每次给出三档润色时，最后都要生成一个静态 HTML 对照页，方便电脑端宽屏比较。若用户明确说"不要生成网页/只在聊天里给"，才跳过。

## 优化目标

优化时从以下三个维度检查：

1. **易懂优先**：读起来顺、信息清楚，不为了显得高级而用绕口的表达
2. **衔接流畅**：句子之间、段落之间过渡自然，不过度衔接也不突兀
3. **文风符合度**：写法是否符合本项目的文风基调（参见 `1-边界/1.2_文风.md`）
优先级：衔接流畅 > 易懂优先 > 文风符合度。三个维度冲突时，以衔接流畅为准。

## 执行前判断

正式润色前，先用一句话判断本段的主要戏剧功能。

### 三档核心差异

**三档不是"删多/删少/微调"的渐变，是"减/调/强"三种不同的执行逻辑。** 每档必须体现不同的操作模式：

| 档位 | 核心方向 | 操作重心 |
|------|----------|----------|
| **简洁** | 做减法 | 偏删减提纯，去掉多余解释、重复信息、弱形容、拖慢节奏的连接和不必要反应。可以重排句子，让表达更干净，但不得删掉必要剧情信息、关键情绪落点或伏笔。 |
| **调整** | 做调整和补充 | 在保留原文风格和剧情事实的前提下，允许补充信息和词语、替换弱词、拆并句子、调整分段、微调句序、压掉多余解释、强化节奏落点，让因果、判断链、情绪反应、冲突后果或段尾落点成立。 |
| **强化** | 做结构性重写式强化 | 核心目标是让文本更易懂、更顺、更有网感，并让段落功能被充分写出来。允许在不改变剧情事实、人物关系、关键信息和伏笔方向的前提下，重排镜头顺序、重建段落节奏、补足动作链、情绪链、判断链、环境反馈、冲突后果和段尾余味。强化版不能停留在句子润色或少量补词，必须体现比调整版更明显的结构推进和场面塑形；但也不是堆辞藻、堆解释或机械拉长篇幅。

### 生成方式

三档必须独立生成，每档都从**原始原文**出发，不得在前一版结果上修改。

每档写完当前版本后，必须回到原文重新读取，再写下一档。

---

## 段落功能诊断

润色前先判断本段主要功能，可多选：

1. **推剧情**：事件是否更清楚、更有因果？
2. **转场**：空间、时间、气氛变化是否清楚？
3. **压迫/冲突**：威胁、服从、反抗是否有力度？
4. **人物塑造**：角色身份、性格、状态是否更鲜明？
5. **关系推进**：人物之间的试探、误会、信任、敌意是否更可读？
6. **伏笔/破绽**：异常线索是否具体可见，但没有提前揭谜？
7. **情绪落点**：这一小段最后留给读者的感觉是否明确？

只补足本段实际承担的功能，不强行给所有段落添加所有功能。

## 具体策略

### 策略零：原文形态保留

润色前先判断原文形态。

默认输出应像"作者自己精修后的版本"，而不是"AI重新描写后的版本"。简洁版和调整版优先保留原文形态；强化版可以突破原文句序和分段形态，但必须保留原文的剧情骨架、叙事视角、人物状态和文风底色。

### 策略一：主语检查

读到无主语的执行动作时，检查读者是否需要往回找执行者。如果需要，补上主语。

### 策略二：紧迫感缺失检查

读到一个动作时，如果原文用了中性描述词，且上下文有压迫、服从、危险等信号，检查是否缺了强度匹配的副词或动词。

### 策略三：状态转折检查

遇到外人离开、门关上、危险解除、身份切换、独处、胜负已分、场景转入安全区等节点，检查人物外在状态是否需要变化。

状态变化可以通过动作、呼吸、坐姿、说话方式、沉默、脱力、强撑、收敛或破绽呈现。不得凭空新增会改变剧情的心理动机。

### 策略四：场景反差检查

如果段落中同时存在两种质感，如冷/暖、明/暗、干净/脏污、贫/富、喧闹/安静、体面/污浊，检查是否需要强化对比。

反差优先通过具体物件、气味、温度、光线、身体状态和动作呈现，不空喊"反差很大"。

### 策略五：人物判断链检查

当角色产生"奇怪、不对劲、怀疑、害怕、愤怒、心动、信任、警觉"等判断时，必须检查读者能否看见判断依据。

判断链优先为：看见的细节 -> 与身份/常理/前文信息冲突 -> 得出判断。

禁止只写结论而没有支撑，也禁止直接替角色解释过多。

### 策略六：情绪反应链检查

当人物被戳中、慌乱、羞恼、暴怒、强撑、松弛、崩溃或恐惧时，不只检查表情，还要检查动作和语言是否配套。

反应链可以从以下类型中选择2-4个最有辨识度的表现：

- 手上动作失控
- 呼吸、咳嗽、停顿
- 耳根、眼神、颈侧、指尖等局部变化
- 话语结巴、转移话题、强行命令
- 视线躲避、身体后撤、坐姿变化

反应必须服务人物性格和当前情境。

### 策略七：伏笔可读性检查

如果本段承担身份破绽、危险预警、关系变化、后文钩子、规则漏洞等功能，必须让线索具体可见。

线索要让读者能察觉异常，但不得提前揭谜。不同场景使用不同细节，禁止套用固定模板。

### 策略八：润句与润场区分

润色分为两层：

1. **语言层**：把句子改顺、主语补清、词语换准、节奏压实。
2. **段落层**：把段落功能写足，包括状态转折、场景反差、人物判断链、情绪反应链和伏笔可读性。

### 策略九：有效改动要求

润色不能等同于保守校对。调整版应体现至少两类有效改动，强化版应体现至少三类有效改动，且至少包含一类段落层或场面层改动：

- **词语更准**：替换虚弱、重复、粘连或不贴场景的词。
- **节奏更稳**：通过拆句、合句、换行或调整句序强化阅读推进。
- **落点更清**：可以补充内容，把动作后果、情绪反应、判断依据或段尾余味表达得更清楚。
- **视角更稳**：减少跳视角、解释腔或不符合人物认知的表达。
- **场面更足**：在不改变剧情事实的前提下，补足可感知的动作、声音、空间、物件、旁人反馈或环境压迫，让读者看得见现场。
- **链条更完整**：把原文里跳过的起因、动作承接、身体反应、心理判断、外部后果或段尾钩子补出来，但不提前揭示后文秘密。

## 不优化清单

以下情况不要优化：

1. **角色专属腔调**：某角色就是说话慢、啰嗦、带废话——那是人设，不是要删的内容。
2. **作者故意留白**：如果当前信息明显应留到后文揭示，不提前解释或明说。

---

## 执行纪律

如果具体策略条目对某项措辞给出了明确的判断方向（如策略二指出"应补强度匹配的副词"），AI 必须按条目方向执行，禁止反向操作；但执行时必须服从当前版本档位，不能为了执行某条策略破坏简洁、调整、强化三档的差异。

## 版本写出后自查（每个版本执行完毕后，输出正式结果前，必须逐条对照）

1. **本版本的操作类型是否匹配其档位定义？**  
   简洁版以"去掉"为主 > 如发现本版本改动以替换/补充为主，说明方向偏了，重写。  
   调整版以"替换/调整/补"为主 > 如发现本版本改动以"去掉"为主，说明方向偏了，重写。  
   强化版以"结构性重写式强化"为主 > 如发现本版本只是补几个词、换几句表达，或改动主要是"去掉/替换"，说明方向偏了，重写。

2. **三个版本的操作类型是否真的不同？**  
   简洁版、调整版、强化版的改动应该体现三种不同的操作方向。如果三个版本的改动让人感觉"只是删多删少的区别"，说明没有执行到位，调整版和强化版中改动量最小的那个必须重写。

3. **强化版有没有比调整版多做结构性补强？**  
   可能的补强维度：镜头顺序、段落节奏、动作链、冲突后果、判断链、情绪反应、身体状态、环境反馈、人群反应、段尾余味、因果衔接。如果强化版和调整版的差异只是换词、补短句或轻微扩写，说明没有真正执行"结构性重写式强化"，强化版必须重写。

4. **强化版是否守住原文事实边界？**  
   强化版可以新增现场细节和承接动作，但不得新增会改变剧情走向的人物决定、关系结论、能力设定、关键道具、明确阴谋解释或后文谜底。新增内容必须服务本段已有功能。

每次润色默认输出三个版本，三个版本代表不同润色层级。

## 静态网页预览页

每次完成三档润色后，必须在项目根目录执行固定脚本，把预览数据通过标准输入喂给脚本，生成或覆盖项目下 `.temp` 目录的 `润色结果预览页.html`（路径：`.temp\润色结果预览页.html`）；不要手写 HTML、不要改样式、不要生成历史文件。若当前工作目录不确定，先切到项目根目录，或使用脚本绝对路径。

```bash
python .agents/skills/writing-optimization/scripts/render_compare_html.py --stdin --output .temp\润色结果预览页.html
```

必须使用 `--stdin`，不要使用 `--json` 命令行参数传整段 JSON；若运行环境已经是 PowerShell，不要再用 `powershell -Command "..."` 或 `pwsh -Command "..."` 包裹整段正文/JSON/命令；直接在当前 PowerShell 上下文里用变量和管道执行。JSON 最小结构：`source`（原文）、`function`（剧情功能诊断）、`versions`（三个版本，每个含 `name` 和 `text`）。`recommendation` 格式：推荐版本标识（从原文/简洁版/调整版/强化版四个中选一个）+ 推荐理由。例：`推荐<strong>调整版</strong>。\n\n理由：<推荐理由>`。如需强调，只允许 `<strong>` 和 `<br>`。必须先在内存中组织 JSON，再直接写到脚本标准输入；不要创建临时 JSON 文件或其他一次性脚本/文件。

**单行执行兼容**（适用于 AI 工具等只能发送单行命令的环境）：
- 将 JSON 存入 PowerShell 单引号字符串变量，JSON 内部换行使用 `\n` 转义序列（Python `json.loads()` 会正确解析）
- 然后通过管道传给脚本：`$json = '{"key":"val\nval2"}' ; $json | python ...`
- 单引号字符串内唯一需转义的是单引号本身（双写 `''`）；JSON 字段值中避免使用英文单引号，或用全角 `「」` 替代

```json
{"title":"润色结果预览页","source":"原文","function":"剧情功能诊断","versions":[{"name":"简洁版","text":"正文"},{"name":"调整版","text":"正文"},{"name":"强化版","text":"正文"}],"recommendation":"推荐<strong>版本名</strong>。\n\n理由：<推荐理由>"}
```

脚本不可用时先说明原因，再手写同等结构预览页。

聊天输出默认只有三行：第一行写 `哪个能点就点哪个，兼容不同编辑器。`，第二行是 Markdown 链接 `[润色结果预览页](.temp/润色结果预览页.html)`，第三行是 inline code 文件名 `` `.temp\润色结果预览页.html` ``。三行后面加一句换行说明：`注意，如果已经打开过预览，可以在原预览页看到最新内容。`不要在聊天里展示"写入脚本/执行脚本/删除脚本/打开网页/执行成功/DONE"等过程信息。原文、三版完整正文、剧情功能、优化点和推荐理由都必须放入 HTML 预览页；页面以"原文 + 简洁版 + 调整版 + 强化版"并排对照。聊天窗口默认不输出正文代码块；只有用户明确要求聊天内正文时，才用代码块呈现，方便复制。
